# Work Order — Automatic Mode

The autonomous variant of [work-order](execute-work-order.md), invoked as `/execute <work-order> automatically` — typically by an Executor running [execute-milestone](../../executor/workflows/execute-milestone.md), and usable directly. It runs **the same sequence** as the manual workflow, which stays the canonical reference; this file states only the overrides, for every work-order type.

## Premise

This mode assumes the plan's design decisions were already confirmed upstream — at the Executor's Reconnaissance checkpoint, or by an operator invoking it deliberately on a vetted document. It trades the manual workflow's interactive sign-off for autonomous convergence, escalating only on genuine novelty. Do not run this mode against an unspecified or un-vetted objective.

## Overrides on execute-work-order.md

Follow every step of [work-order](execute-work-order.md), with these five changes:

- **Step 7 (blessing) runs headless — and it gates.** The Architect Blessing is non-interactive; run it exactly as the manual workflow states (consult-fork the standing handle or fresh fable dispatch, two bounded refinement rounds). If the verdict is not `BLESSED` after the bound, do **not** proceed to implementation: record the unresolved critique in the `## Blessing` section and stop with an `ESCALATE` Acceptance Verdict carrying the Architect's position — autonomous execution never proceeds over an unblessed order.
- **Step 8 (pre-implementation sign-off) → pause-on-novelty.** Do not pause for sign-off; proceed on the confirmed plan. **Pause** — end your turn with the question, context held, truly awaiting — **only** if a genuinely new design decision surfaces during implementation that the confirmed plan does not cover (architecture, data model, security trade-off, scope); the caller answers via SendMessage and you resume in place. **Before returning control, consult the plan's architect** when the work doc's `sessions:` frontmatter carries a consultable planning callback whose `<machine>` segment matches this host — use the **last** `architect@…` entry (legacy forms: `planner@…` annotated `# milestone architect`, or `plan-session:`/`plan-machine:`): `bun .claude/bin/consult "<handle>" "<the specific question; what changed since planning; answer from the plan's intent; flag if this needs a human decision>"` (the required consult path — the script owns the restore-before-resume ladder and the read-only fork boundary; fall back to `--model opus` if fable is unavailable). If the answer resolves the fork as _interpretation of the confirmed plan_ → distill the consult into the work doc and proceed; no escalation. If the architect flags a genuinely new decision, or no consultable session exists → record the question **and the consult's analysis** in the work doc, then **pause**: end your turn awaiting the answer — the caller answers via SendMessage and you resume in place. The same ladder serves **design questions** on an order referencing a design brief: interpretation of the brief/canon → consult the last `designer@…` entry in the brief's `sessions:`; a genuinely new *taste* decision → operator-gated — record the options (screenshots/renders where cheap) and stage it, never guess. Never guess past a real fork — and never treat a consult as license for a new design decision; those belong to the caller. (A headless-fallback Manager cannot pause — its ended turn ends the session. It instead records the question and stops with an `ESCALATE` verdict, the terminal verdict being its only upward channel.)
- **Steps 10–14 (review → resolve → gate → verify activation → inspect) → convergence loop.** Run them as a loop against the order's **target criteria** — *feature/chore*: the Feature Request's acceptance criteria; *bug*: the verification criteria (the reproduction no longer reproduces); *refactor*: the objective's stated invariants + no regressions — not once. Each pass: dispatch the reviewer (and, on `interface` scope, the inspector — which also surfaces UX gaps, not only pass/fail); classify each finding **blocking** or **non-blocking**; resolve blocking findings via the implementer; re-evaluate against the criteria. Continue while blocking findings remain and the **pass budget** (default 3) is unspent. Stop on convergence (a pass yields no new blocking findings), on stall (a blocking finding recurs unresolved → escalate), or on budget exhaustion (→ escalate residuals). Fold non-blocking findings into a pass only while budget remains; otherwise record them as deferred.
- **Quiet running.** Suppress the interactive status narration and step 19's optional investigation offer. Still file out-of-scope bugs (standing behavior), still run the quality gate, still group commits logically with the standard prefixes + Co-Authored-By footer — the close-out commits still fire before the verdict is written (`work/README.md` § The follow-through rule).
- **Progress heartbeat (crash-safe ledger).** Append one timestamped line to a `## Progress` section of the work doc at each step transition, so any interrupted Manager's state survives — crash, cancellation, or early stop — and the Executor (or a re-entry per the manual workflow's Entry state) can read where-in-the-workflow it is without a transcript. Create the section if absent. Log at: planner dispatched (step 5); blessing verdict, with round count (step 7); implementation started (steps 8–9); each implementer phase landed, with the commit sha; each review/resolve pass (pass N — findings count, resolving/converged); quality gate result (step 12); Acceptance Verdict written. Format: `- <HH:MM> <event>` (e.g. `- 14:03 review pass 2: 1 blocking finding, resolving`). Transition points only — the ledger is greppable state, not narration.

## Execution discipline

Process-hygiene rules for any autonomous Manager driving external work. The first binds every role.

- **Exact PIDs only (universal).** Never `pkill`/`killall` on a name or path substring — any session whose prompt embeds a path can match its own command line and kill itself. Kill exact captured PIDs, or port-derived PIDs (`ss -tlnp`), only. This rule binds every role — carry it into specialist briefs for process-touching phases.
- **Foreground or detach — choose deliberately.** An external process that fits under the Bash tool ceiling runs as a foreground block; one that doesn't is detached via `setsid` with the PID and log path captured into the work doc.
- **Never background-and-wait.** A turn that ends "waiting" on backgrounded external work ends the session with the work ownerless; a detached long-runner's finalization is staged for the caller (the Executor), never awaited.
- **The final-mile exception (bounded).** On an explicit Executor instruction to finish convergence (e.g. a verdict-absence nudge), the Manager may complete **mechanical remaining convergence steps** by direct tool calls — the narrow exception to "don't implement directly": mechanical remaining steps only, under explicit Executor instruction, never self-invoked.

## Acceptance Verdict (close-out)

In place of the manual close, append a final `## Acceptance Verdict` section to the work doc — the structured handback the caller reads. The verdict is written **after** the close-out commits land, never before them (`work/README.md` § The follow-through rule): a handback never reports done over uncommitted work.

- `PASS` — all target criteria met, blocking findings cleared.
- `PARTIAL` — criteria met except an enumerated residual list, each line with a reason.
- `ESCALATE` — a genuinely terminal stop, enumerated: the convergence loop's stall or pass-budget exhaustion (residuals listed), an unblessed order after the refinement bound, or an operator-gated stop the caller must action. Interpretation questions are never `ESCALATE` — they pause mid-flight per the Step-8 override (headless fallback excepted, as stated there).

List any criterion that requires a deploy, live runtime evidence, or empirical measurement under the verdict as `operator-gated` — staged for the operator, not counted as a blocking finding. Type-specific close-out rituals still apply (*bug*: the `resolution:` frontmatter + `## Resolution (<date>)` section). The work doc otherwise accumulates the same `<h2>` sections as the manual workflow, plus this verdict.
