# LAN intercom — a repo-scoped consult server + git-native parallel work across machines

> Proposal, 26-07-22, from an operator dialog. Operator framing that seeded it: three LAN machines resolvable via mDNS (`play.local`, `antec.local`, `benpc.local`); "a super simple Bun server that exposes a service port," secured by a shared key and LAN-only firewalls; multiple machines "working potentially on the same branch … running work orders (especially one-off bug fixes) in parallel"; "like a claude consult line"; possibly "the codebase-memory AST (subgraph of touched files) so that the agents are capable of working carefully without stepping on one another's toes"; and the question — "would this require a dedicated 'coordinator' agent?"

## The concept

This realizes the **agent-service API rung** that [`cross-lan-agent-calls`](26-07-05.agentic-workflow/26-07-03.cross-lan-agent-calls.md) designed and deferred — arriving ahead of the runner framework it was gated behind, and **superseding that proposal's SSH lean** on a security determination (below). It also composes the cross-machine *work* story the transport question left open: parallel work orders across machines coordinated through git and the work documents themselves, with the codebase-memory graph supplying declarative conflict avoidance.

Two principles govern the whole design:

1. **The intercom carries judgment; git carries state.** Machines consult each other over the wire ("does your plan touch the scheduler too?"); they coordinate claims, progress, and results exclusively through pushed work documents. No control flow rides the intercom.
2. **Authorize one operation, not a shell.** The SSH rung was too liberal: a generic shell exposes the whole remote system. The intercom authorizes exactly one thing — a **read-only resume-fork within the shared git repo** — shrinking the trust surface to what the invite-only repo already governs.

## The transport — a per-machine Bun consult server

A small Bun HTTP server per machine (`.claude/bin/` tooling, sibling of `consult`; it graduates to `src/` if it grows into the deferred runner framework):

- **`POST /consult`** `{repo, handle, question}` → validate key → resolve `repo` against the machines' registry row (repo path per machine — the column the identity proposal added) → refuse anything not registered → `cd <repo-path> && claude -p --resume <session-id> --model <derived> "<question>"` → stream the answer back.
- **`GET /health`** → liveness + version. Transport diagnostics only — *not* a work-status channel (see Coordination).
- **Client seam**: `.claude/bin/consult` already stops with a clear message on a machine-mismatched handle — the reconciled plug-in point. On mismatch it now resolves the handle's `<machine>` to `<machine>.local` via the registry (new **host column**) and POSTs. Nothing above the transport changes; the callback grammar carries the routing key by design.
- **Model derivation — token conservation.** The handle's role segment implies the session's model via the existing pins: `architect`/`planner` → fable, `manager`/`executor` → opus, specialists → sonnet. The server derives `--model` from the role instead of hardcoding fable (the current consult recipes' default), matching the called-back session's tier. Fall back one tier down on unavailability, per the standing rule.
- **Security posture** (operator-confirmed):
  - **Shared bearer key, required** — an untracked env file per machine. The endpoint is remote prompt execution against full-context sessions; the key is the first layer, not an option.
  - **LAN-only firewall** on the service port — defense-in-depth, second layer.
  - **Repo-scoped authorization** — the server serves only registered repos, `cwd`-pinned; remote machines introspect the shared project, never the host system.
  - **Read-only forks** — the server dispatches every resume with a no-writes permission profile. Doctrine (consults are advisory, never authority) enforced at the transport: a confused or compromised consult can leak reasoning, not act.
- **What SSH-exec footguns this retires**: the non-interactive-PATH class and the Windows OpenSSH quoting fork (benpc) — the server is a login-context process that owns its own environment.

## Parallel work across machines — git-native, stigmergic

Across machines there are no shared worktrees — **each machine has a clone**. Coordination is through durable artifacts:

- **Claiming**: a machine claims a work order by stamping `claimed-by: <machine>/<session-id> (<timestamp>)` in its frontmatter and pushing. A rejected push means another machine claimed first — **push-conflict-as-lock**, git's own atomic primitive, no arbiter in the loop. (This adopts the git-workflow doc's runner claim-protocol, which cross-lan flagged for co-adoption.)
- **Blast radius, stamped at claim**: the claiming Manager computes the order's file-level blast radius from the codebase-memory graph (`search_graph`/`trace_path` over the plan's symbols → file set) and records it in the order (a `blast:` frontmatter list + the commit it was computed at). Conflict avoidance becomes **greppable state**: before claiming, check the radius against every live claimed order's recorded radius.
- **Branch policy** (operator-confirmed): **shared branch on disjoint radii** — both machines commit to the branch (pull–rebase–push); on overlap the later claimant defers, or takes branch isolation for that order with the **Integrator** owning the merge. Radii are advisory (plans drift) — the reviewer and Integrator still guard the merge regardless.
- **Stale-claim rule**: a claimed order whose `## Progress` ledger hasn't advanced in ~N hours (judged at fetch time from the ledger's own timestamps) is presumptively stalled; the claim may be contested and released — the cross-machine analog of the Executor's mtime-stall rule. Operator adjudicates contested releases while the work-queue tier is deferred.

## Coordination — no coordinator, no pinging

Settled in the seeding dialog, recorded as determinations:

- **No dedicated coordinator agent.** It would be a second source of truth (documents are the durable state; claims/radii/BACKLOG are greppable by any session on any machine); the only atomic primitive needed is push-conflict-as-lock; and the role that owns queue-level judgment already exists as the deferred **work-queue tier** (`_deferred/26-07-11.work-queue-tier.md`) — the Operator plays it today. When that tier lands, this server is the natural home for its dispatch endpoint; that is a separate, later adoption.
- **No automatic work-status pinging.** Status already rides git — `claimed-by:`, the `## Progress` heartbeat ledger, Acceptance Verdicts — so **`git fetch` is the ping**: pull-based, on-demand, zero standing cost. A machine working alone pays nothing; a machine asleep breaks nothing. Push-based pings would buy a liveness protocol (intervals, timeouts, sleep false-positives) to answer questions the ledger answers with timestamps. `GET /health` exists for transport diagnostics only.

## Boundary and non-goals

- **Not a dispatch bus, not a cluster scheduler.** Consults only. Dispatch across the queue belongs to the deferred work-queue tier.
- Studio-LAN scale: a handful of trusted boxes behind mDNS; Tailscale/relay extends the same routing later without changing the grammar (cross-lan's future note stands).

## Depends on / enables

- **Depends on** (all live): the machine-qualified callback grammar + registry (`work/README.md`), the durable archive (`work/sessions/` — restore-before-resume works cross-machine), `bin/consult`'s ladder, and the codebase-memory graph (BASELINE) for radii.
- **Supersedes** the SSH rung of [`cross-lan-agent-calls`](26-07-05.agentic-workflow/26-07-03.cross-lan-agent-calls.md) (reconciliation note there points here); its API-rung design and machine-dimension reasoning carry forward.
- **Enables** the work-queue tier's transport, and multi-machine campaigns generally.
