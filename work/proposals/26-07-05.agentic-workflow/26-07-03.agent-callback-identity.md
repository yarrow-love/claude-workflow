# Agent callback identity — a role-qualified provenance stamp for the call chain

> Proposal, 26-07-03. Operator insight that seeded it: the call hierarchy (mission-planner → milestone-planner → milestone-executor → objective-orchestrator → task-specialists) is enabled "simply by providing the session id in the frontmatter." The proposed upgrade: an **agent list** — each agent in the chain that had a hand in originating the document — under a terse, machine-qualified grammar: `agent: - <role>@<machine>:<session-id>[:<message-id>]` (e.g. `mission-planner@play:86ade408-7a9b-4edd-991a-6b3592ce377a`).

## The concept

This is the **general form** that [`planner-callback`](26-07-03.planner-callback.md) proved at n=1 and explicitly deferred: "a role→session-lineage registry — every long-lived context registered and consultable by role name rather than raw id." Where planner-callback records a single `plan-session:` / `plan-machine:` pair for one tier, this generalizes it to **every tier**, as one field: the document's `agent:` list _is_ the call chain that produced it.

Each entry is a **callback handle** — because `--resume <session-id>` forks a session at full context (planner-callback's enabling fact), any tier can dial any ancestor in the list and consult the agent that authored a decision, on its own machine. The list is also a **provenance record**: "who decided this, and can I ask them why?" becomes greppable.

## The grammar

```
<role>@<machine>:<session-id>[:<message-id>]
```

- **role** — the tier/agent name (`mission-planner`, `milestone-planner`, `executor`, `orchestrator`, and — open question below — specialists). Consultable by role, not raw id.
- **machine** — the host that owns the session transcript (`play`, …). Load-bearing: transcripts are machine-local (planner-callback's Limits), so the machine is the routing key a cross-machine consult needs — see [`cross-lan-agent-calls`](26-07-03.cross-lan-agent-calls.md).
- **session-id** — the fork target for `--resume`.
- **message-id** _(optional)_ — a reference pointer into the transcript for provenance precision. Not a resume target (resume forks from the session, not a message); omit unless a specific turn needs citing. This document's own stamps omit it.

## Why unify

Today the callback is captured in three disjoint places: `plan-session:`/`plan-machine:` (planner-callback), the `session_id` in the campaign log (introspection), and nothing at all for the mission/orchestrator tiers. One terse, ordered, machine-qualified `agent:` list makes the whole ladder consultable and auditable from the artifact itself — no spelunking three conventions.

## Compatibility — additive, not a rip-out

Per the operator's framing ("keep the convention the same for now, but consider how we may upgrade"): `agent:` sits **alongside** the existing `status:` / `spawned:` / `plan-session:` frontmatter. Nothing migrates mid-flight. `plan-session:` + `plan-machine:` collapse into a single `agent:` entry _over time_, as tiers adopt it. This proposal and its two siblings carry the stamp as the first dogfood — provenance for the design discussion that produced them.

## The boundary that keeps it honest

A callback is **provenance + a consult handle, not authority** — the same interpretation-vs-decision line planner-callback drew. An `agent:` ancestor answers "what did you intend / why was X rejected"; it does not own _new_ design decisions, which still escalate to the human. The list makes the chain _reachable_; it does not make any link _the boss_.

## Open forks (for the adopting work order to settle)

- **Field name** — `agent:` (operator's term) vs `agents:` (reads as a list) vs `lineage:`. Lean: honor `agent:`.
- **Who gets an entry** — only durable, resumable-session tiers (planners, executor, orchestrator), or also transient specialists (a reviewer subagent has an intra-session id, not a `claude -p` session; SendMessage-by-id is its callback). Lean: durable tiers in `agent:`; specialists stay intra-session.
- **Machine short-name registry** — `play` is terse and stable; the studio needs a small canonical map so names don't drift per box. A one-liner in `docs/git-workflow.md` or the mission charter.
- **message-id semantics** — reference-only, or ever a resume anchor if the CLI grows message-addressable resume.

## Reconciliation note (26-07-03, full-context review)

Aligned with the adopted planner-callback's deferred "role→session-lineage registry" — this is that registry as a frontmatter-native convention, which is the right substrate (greppable, zero infra, travels with the artifact). One addition to the machine registry it proposes: carry each machine's **repo path** and shell flavor alongside the short name — the cross-LAN proposal's remote-resume mechanics need both (see its reconciliation notes).

## Spawn plan (when adopted)

A `todos/` work order: define the grammar in one place (the git-workflow doc or a short `docs/agent-identity.md`), add the `agent:`-stamp close-step to the planner/executor/orchestrator workflows (each records its own entry at close, prepending to the chain it inherited), and fold `plan-session:`/`plan-machine:` into it. Gated behind [`cross-lan-agent-calls`](26-07-03.cross-lan-agent-calls.md) only for the _cross-machine_ consult path; the single-machine provenance stamp is adoptable alone.

## Reconciliation note (26-07-11, astrology-repo determinations — adopted as amended)

Adopted by the [determinations record](../../todos/_done/agentic-workflow/1.determinations.md), with the open forks resolved **against** two of this proposal's leans:

- **Grammar**: `<role>@<machine>/<session-id>[#<message-id>]` — the locked *slash* form, not this proposal's colon form; the optional message-id rides a `#` fragment (URL semantics: a citation into the transcript, greppable in the archive, **not** a resume target).
- **Field name**: `sessions:` (the locked 26-07-05 convention), not `agent:` — one field for both the originating chain and subsequent touches, append-only.
- **Who gets an entry**: durable resumable tiers, per this proposal's lean (confirmed).
- **Machine registry**: lives in [`work/README.md`](../../README.md) (machine, OS/shell, repo path), per the sibling proposals' reconciliation notes.
