# Accomplish live feed — a jsonl→human renderer + tee dispatch for real-time introspection

> Proposal, 26-07-03. Operator insight that seeded it: "the system ran all night — wonderful — but the Orchestrators' background shells still show no output. I cannot see what is going on behind the scenes, so I cannot nudge the Orchestrator when I notice drift." The introspection upgrade (landed 26-07-03) fixed the _wire_; this fixes the _window_.

## The gap the introspection upgrade left

The introspection work order switched the phase-Orchestrator dispatch to `--output-format stream-json --verbose > tmp/accomplish/<phase>.jsonl 2>&1` — the events now stream to a growing file the operator _can_ tail. But two things still blunt live introspection:

1. **The background shell stays silent by design.** Output is redirected into the `.jsonl`; watching the shell (or the Bash-tool background task) shows nothing. The file is the feed, not the shell.
2. **Raw `stream-json` is not human-readable.** The documented `tail -f … | jq -r 'select(.type=="assistant") | .message.content[]? | .text'` one-liner is a workaround, not a window — no tool-use context, no `## Progress` transitions, no cost/verdict summary.

The `scripts/accomplish-tail.ts` renderer was scoped into the introspection order and then **cut as "optional."** Its absence is exactly what the operator is now feeling.

## The design

**`scripts/accomplish-tail.ts` — jsonl → legible progress lines.** Reads either a file (follow mode, `tail -f` semantics) or stdin (pipe mode). Per event type: assistant text verbatim; tool-use as a one-line `→ <tool>(<terse args>)`; the `system/init` line as a header (session id + model); the terminal `result` as a cost/duration/verdict footer. Tolerates unknown `.type` values (a `rate_limit_event` was observed mid-stream) — unknown lines are skipped, never fatal. Bun script, `bun run scripts/accomplish-tail.ts <file>` (repo toolchain footgun: `bun run`, not bare node).

**The dispatch upgrade — tee through the renderer.** For an operator-launched campaign:

```sh
set -o pipefail
claude -p "/build <phase> automatically" --model opus --permission-mode acceptEdits \
  --output-format stream-json --verbose 2>&1 \
  | tee tmp/accomplish/<phase>.jsonl \
  | bun run scripts/accomplish-tail.ts
```

The background shell becomes a **live, human-readable feed**, and the raw `.jsonl` still persists on disk for the Executor's bounded `tail -c` check-ins. The two consumers are complementary: the human reads the rendered feed; the Executor reads the file.

**The one real trap — exit code.** A pipe's exit status is the _last_ command's; the Executor must still read the terminal `result` event (verdict/cost) from the **tee'd file**, not the pipe, and `pipefail` must be set so a `claude` non-zero still surfaces. The renderer never affects control flow — it is a pure view.

## Relation to the landed work

This is the ergonomic front-end to the introspection order — same `.jsonl`, same `## Progress` ledger, same `session_id`-at-launch. Nothing about the Executor loop changes; the renderer is additive. It is the piece that turns "the system ran all night" into "I watched it run all night and steered it twice."
