# Durable resumability — archiving the transcripts behind the `sessions:` handles

> Proposal, 26-07-06. Operator insight that seeded it (M2-architect forensics debrief): *"these callbacks live only as long as the transcripts do — `~/.claude/projects/` is cache-like, not archival. If callback-ability matters long-term, the transcripts referenced in `sessions:` frontmatter deserve a backup path."* The callback system's whole value is reaching an author **at full context**; that context is one unmanaged directory away from vanishing.

## The problem — handles pointing into a cache

A `sessions:` entry (`role@machine/<uuid>`, the locked 26-07-05 convention) is a **pointer into `~/.claude/projects/<cwd-slug>/*.jsonl`** — a store no fidelity rule governs. Four rot vectors, three already observed at day 3 of the convention's life:

1. **CLI retention.** Transcripts are subject to the CLI's cleanup period (configurable, defaults on the order of weeks). A mission outlives it; [`planner-callback`](26-07-03.planner-callback.md) names this limit and punts ("degrade gracefully").
2. **Machine loss.** `~/.claude` sits in no backup or sync path — the repos synchronize across studio machines; the transcripts do not. A dev-box rebuild silently voids every `@play` handle in the record.
3. **Cross-machine opacity.** The M3 campaign relocated to the studio host mid-run — its transcripts live on `benpc`, unreachable from `play` even while perfectly intact. The handle's `machine` segment routes correctly, but nothing guarantees the target store's contents.
4. **Cache-like treatment by tooling.** The 26-07-04 bulk mtime-touch across unrelated transcripts (found during the M2-architect investigation) shows local tooling already treats the directory as disposable state. The M2 architect was recoverable *only because* the transcripts happened to survive four days — forensics on a cache is not a durability strategy.

## The principle

**The transcript is the artifact behind the handle — extend the M0.5 fidelity rule to it.** A closed session's transcript is immutable by nature, which makes it exactly the kind of object the project already knows how to keep: content-addressed, hash-manifested, copied to a durable home. Stamping a `sessions:` entry into a committed document is a *promise of consultability*; archiving is what makes the promise true.

## The design

1. **Archive-on-stamp.** The moment a workflow stamps a `sessions:` entry into a lifecycle document (the kickoff/close steps that already exist in `accomplish-goal.md`, `plan-milestone.md`, the automatic workflows), it also archives its transcript: `gzip` the `.jsonl` → copy to the durable store as `sessions/<machine>/<uuid>.jsonl.gz` + a `sha256` manifest line. Long-running sessions re-archive at close (last write wins; the transcript only grows). The stamped set *is* the archive set — durable tiers only (architects, planners, executors, orchestrators), exactly the sessions the convention says to record; specialists never stamp, never archive.
2. **The store — forks, with a lean.** (a) *In-repo* (`_/sessions/` or a mission-tier dir): rides existing git sync + backup, zero infra — but transcripts run to tens of MB (gzip ~10:1) and the repo bloats with every campaign. (b) **Central store on the studio workhorse** (`benpc`) — a `sessions/` directory HTTP-served beside the music/lasers services (GET/PUT, bearer-gated like the mutating routes): no repo bloat, one canonical home, and squarely aligned with the M0.5 determination that *the workhorse is the central resource* (the library's sibling, not a new idea). (c) A dedicated archive repo / LFS: heavier than (b) for no gain at studio scale. **Lean: (b)**, with (a) acceptable as an interim for mission-tier sessions only. Whatever the fork, the store itself joins the operator's normal backup discipline — an archive nobody backs up is the same cache one layer up.
3. **Restore-before-resume.** The consult step (accomplish-goal 7.3 and kin) grows a ladder: transcript present locally → `--resume` as today; absent → **fetch from the archive into `~/.claude/projects/<slug>/`** → resume; archive miss → the existing planner-callback degrade (documents + fresh dispatch). One wrapper script (`bin/consult <role@machine/uuid> "<question>"`) owns the ladder so workflows stay one-liners.
4. **The slug problem (real, solvable).** Restore targets a *cwd-slug* directory — which encodes the repo path, and differs per machine (Windows paths on `benpc`). The machine registry that [`agent-callback-identity`](26-07-03.agent-callback-identity.md) proposes (and whose reconciliation note already adds a repo-path column for [`cross-lan-agent-calls`](26-07-03.cross-lan-agent-calls.md)) carries exactly the mapping restore needs: `machine → repo path → slug`. One shared deliverable serves all three proposals.
5. **Local retention floor.** Raise the CLI cleanup period on studio machines so the *local* cache outlives a campaign — a mitigation that buys time for archive-on-stamp to fire, never the durability answer itself.
6. **Mission-close sweep.** The mission-tier protocol ([26-07-02.mission-tier.md](26-07-02.mission-tier.md)) gains one close-step: grep the mission's artifacts for `sessions:` entries, verify every referenced transcript exists in the archive, backfill any stragglers. The frontmatter convention makes the referenced set greppable by design — the sweep is a loop over `git grep -h '@.*/' -- '*.md'`.

## Boundaries

- **A restored session is a consult, not an authority** — the same interpretation-vs-decision line both siblings draw; durability changes *reachability*, never the escalation ladder.
- **Context summarization** on very long resumed sessions is acceptable: consults ask for *intent*, which summaries preserve; anything needing fresh code truth goes to a researcher regardless.
- **Scale**: a campaign archives a handful of transcripts at single-digit MB gzipped — trivial against the stems already stored per track.

## Relation to the siblings

[`planner-callback`](26-07-03.planner-callback.md) (adopted) built the consult channel and *named* the retention limit — this resolves it. [`agent-callback-identity`](26-07-03.agent-callback-identity.md) makes the chain addressable — this makes the addresses *durable*. [`cross-lan-agent-calls`](26-07-03.cross-lan-agent-calls.md) routes **live** consults to the owning machine — the archive is its **offline complement**: a decommissioned machine's sessions remain consultable by restoring its transcripts anywhere.

## Spawn plan (when adopted)

A `todos/` work order: the `bin/session-archive` + restore-aware `bin/consult` wrappers; one "stamp ⇒ archive" line added to each workflow's existing stamp step; the machine registry with repo-path column (shared deliverable with the two sibling proposals); the retention-period config bump on studio machines; the mission-close sweep line in the mission-tier protocol; the store stood up per the fork chosen at adoption (lean: `benpc` HTTP store beside the existing services).

## Reconciliation note (26-07-11, astrology-repo determinations — adopted; store fork resolved differently)

Adopted by the [determinations record](../../todos/_done/agentic-workflow/1.determinations.md); mechanics land via [`agentic-workflow/5.provenance.md`](../../todos/_done/agentic-workflow/5.provenance.md). The store fork resolves to **(a) in-repo** — `work/sessions/<machine>/<uuid>.jsonl.gz` + `MANIFEST`, `git mv` into `work/missions/<v>/sessions/` at mission close — because this repo is **single-machine**: the central-workhorse lean (b) was a lasers-studio answer with no analogue here, and in-repo rides the operator's existing git sync/backup discipline with campaign-scale volumes (single-digit MB gzipped) that don't threaten repo weight. The design gains one command this proposal didn't name: **`/archive`**, the operator checkpoint sweep for missions that are inactive but not closed (interactive sessions never hit a workflow close-step, so archive-on-stamp alone would miss the highest-value design dialogs). The slug problem is trivial at one machine; the registry (now in `work/README.md`) carries the repo-path column regardless, for the cross-LAN future.
