# The mission tier — `work/` as the container for proposals, todos, milestones, and mission archives

> Proposal, 26-07-02, from the workflow-design sessions ([agency-workflow-design](26-07-02.agency-workflow-design.md) is the companion). **No mid-version reorganization** — this specifies the shape and the protocols; the migration executes at the v0.5→v0.6 boundary as part of the mission-close protocol it defines. Operator framing that seeded it: "each time I have started a new version and stashed work docs (e.g. `_/milestones.v0.4`), I have implicitly created a kind of container that _should_ carry its work system with it."

## The concept

A **mission** is the container above the milestone: one version-scale objective (`v0.5 — Time & Rhythm`), its settled design, its milestone roadmap, its work orders, its proposals, and — critically — its **closing report**. The tier has been implicit all along: the version branches _are_ missions in disguise (v0.5 has an objective, a settlement phase (M0), a phased roadmap, and will have a close-out). Naming the tier lets protocols attach to it.

This completes a self-similar ladder that now runs the full height of the system, mirroring the product's own (`primitive → cue → scene → show`):

| Tier | Unit | Planning artifact | Closing artifact | Automation |
| --- | --- | --- | --- | --- |
| work order | one feature/fix | `## Implementation Plan` | `## Summary`, `_done/` filing | `/build` (inner loop) |
| milestone phase | one campaign | phase doc `## Acceptance` | `## Addendum` | `/accomplish` (outer loop) |
| milestone | one capability | `index.md` + phase docs (`/plan <milestone>`) | index status + campaign log | `/accomplish <milestone>` |
| **mission** | **one version** | **charter (`work/mission.md`) + settlement (the M0 pattern)** | **closing report (`closing.md`)** | **execution: none — human-cadenced; close: `/complete <mission>`, operator-invoked only** |

The self-similarity is what keeps the system intelligible: every tier plans, works, and closes the same way, at its own scale. If one-off todos earn work docs and closing summaries, milestones and missions certainly do.

## The layout (at the next mission boundary)

```
work/
  mission.md            # the ACTIVE mission's charter (see Protocol)
  proposals/            # pre-decision design thinking (this directory, relocated; + _deferred/, _rejected/)
  todos/                # active work orders (+ _done/, bugs/ — unchanged internally)
  milestones/           # the active mission's phased roadmap (ex docs/milestones/)
  footguns/             # the ACTIVE mission's footgun ledger (ex docs/footguns/ — see Position 3)
  runbooks/             # the ACTIVE mission's operational runbooks (ex deploy/, src/services/*/deploy/ — see Position 3)
  missions/             # closed missions, archived WHOLE and IN-REPO
    README.md           # the mission ledger: every mission, status, one-line outcome
    v0.4/
      mission.md        # charter (reconstructed for pre-protocol missions)
      closing.md        # the closing report
      milestones/  todos/  proposals/  footguns/  runbooks/
```

The double duty is the point: one directory is both the live work system and the accumulating institutional memory, and mission close is a `git mv` from the first role into the second. `docs/` is thereby purified into pure **knowledge** (footguns, research, lux reference, beyond-sdk) with all **process** under `work/` — a clean noun split the current layout blurs.

## Position 1 — Mission = branch (and this closes a loop with the git-workflow policy)

Every mission gets its own git branch (`v0.5`, `v0.6`), which is exactly the "version branch as shared trunk" of the git-workflow work order (`todos/26-07-02.git-workflow.md`) — the mission tier is what gives that branch semantic identity. Branch open = mission open; the branch's final state carries the mission's archive; mission close tags it.

## Position 2 — Work-doc archives live in-repo (revising the earlier tags-only position)

The agency-workflow proposal's Position 5 said: raw work-doc record → git tags, distilled wisdom → in-tree ledgers. **The operator's rebuttal is accepted, and it's the stronger position:** finished work docs are not raw record — they _are_ distilled design reasoning (research findings, plans, review briefs, phase addenda dense with decisions that never fully reach the footgun ledger). They are arguably more valuable than the stale `src/experiments` code they accompany, and — decisively — **agents grep the working tree; no agent spelunks `git show v0.4-docs-final:…`**. An archive that is never consulted is not memory. The v0.4-footgun files exist precisely because that wisdom kept being needed; `work/missions/<v>/` generalizes the mechanism from curated excerpts to the full corpus, namespaced so it never pollutes active-work paths.

Tags remain right for what they were right for: marking the close commit, and burying _truly_ dead material (abandoned drafts, superseded scratch). The `_/<stash>` pattern retires entirely.

## Position 3 — Footguns (and runbooks) are per-mission

Footguns belong to the mission's work system, not to the project eternally. The discovered footguns of one mission are often _precisely the cause_ of the next mission's redesign — and when the new design lands, those entries describe traps that no longer exist (exactly what happened at v0.4→v0.5: the redesign dissolved whole subsystems whose footguns then had to be hand-sorted into "archive" files). Packing them into project-scoped documents (root CLAUDE.md, README) is how staleness accumulates; the mission boundary is the natural refresh line.

**Protocol — footgun triage at mission open (part of settlement):** walk the predecessor's ledger and disposition every entry:

- **designed-around** — the new mission's design removes the trap by construction; it stays in the archive as the _reason_ for the design (source material for critical thinking, exactly as the operator framed it);
- **carried** — still true; moves into the active `work/footguns/` ledger, adapted to the new design's terms where needed;
- **watch** — relevance unknown until the subsystem is touched; noted in the active ledger as a pointer, resolved on first contact.

**Runbooks ride the same rail.** Operational procedure docs currently stray across the tree (`deploy/RUNBOOK.md`, `src/services/lasers/deploy/RUNBOOK.md`, `src/services/music/deploy/RUNBOOK.md`, `docs/runbooks/`) — they consolidate at `work/runbooks/<slug>/` (the procedure plus its operator-applied config payload, e.g. a Caddyfile, bundled per runbook). A runbook describes _deployed state_, which is mission-versioned in practice: a redesign obsoletes procedures exactly as it obsoletes footguns, so the same open-triage applies — **carried** (the system is still deployed; the carry re-certifies the procedure), **retired** (the system it operated is gone; it archives as the record of how that system _was_ run), **watch**. Migration nuance: build-referenced deploy _assets_ (a `public/` a service serves at runtime) may stay in-package; what moves is the operating procedure and the config an operator applies by hand.

**The act of carrying is itself the freshness review.** Durable entries (the `bun --bun` toolchain trap) simply get carried every mission — cheap, and each carry re-certifies them. CLAUDE.md's footgun index always points at the active `work/footguns/` only, so todos, milestones, proposals, _and_ footguns are **always relevant to the active mission**, while the full record persists in `work/missions/<v>/footguns/`. (Today's extraction of the ledger to `docs/footguns/` set this up unknowingly: the live files become `work/footguns/`, and the `v0.4-*` archive files are already the shape of `work/missions/v0.4/footguns/`.)

## Position 4 — The mission protocol (charter + closing report)

**Charter — `work/mission.md`, written at mission open.** The objective and why-now; scope and explicit non-goals; the settlement reference (a v0.5-M0-style determinations phase is the _opening move_ of a mission — that pattern is now protocol, not accident); success criteria; and **failure criteria** — what observed state would mean this mission should close unaccomplished. Naming failure conditions up front is what makes failure a legitimate outcome instead of a quiet abandonment.

**Closing report — `work/missions/<v>/closing.md`, written at mission close.** Outcome (`accomplished | failed | superseded`); what shipped vs what didn't and why; the DX retrospective (what the _work system itself_ taught — this session's model-tier/planning/introspection work is exactly the kind of learning a v0.4 closing report would have carried); items carried forward (open todos triaged: forward into the next mission or archived with cause); and a **reader's index** — the handful of archived docs a future planner should actually read, so the archive has a front door and the distillation gate survives the volume.

**Failure is first-class.** A failed mission closes with the same ceremony: report written, branch preserved, corpus archived, lessons indexed. This is what licenses ambitious missions — the deictic-reference/conversational-inference system (explicitly out of scope for lasers, per 26-07-02 decision) is the archetype of a future mission that _deserves_ a charter with honest failure criteria.

**Lifecycle:** `proposed (a proposal here) → chartered (mission.md + branch) → settled (determinations + footgun triage) → active (milestones execute) → closed (via /complete — report + archive + tag)`.

## Position 5 — No `/accomplish <mission>`, by design

The temptation is named and declined, on the operator's own reasoning: serial milestones must be **sharpened by what the previous milestone taught**. The inter-milestone gap is where phase addenda, footgun reconciliation, and the campaign log feed the next `/plan <milestone>` dialog — automating across it would launder away exactly the learning the mission tier exists to accumulate. The mission is human-cadenced; its artifacts (charter, closing report, ledger) are what make that cadence _cumulative_ rather than merely manual. **Preliminary objective planning per milestone at settlement is fine and is already practice** — v0.5's roadmap laid gist-level milestone stubs (index scope + phase gists) up front; what stays just-in-time is the _hardening_ (`/plan <milestone>` filing the phase docs), because sufficient milestone planning usually depends on prerequisite outcomes. The command family: charter + settlement at open (the M0 pattern, stubs included) → per-milestone `/plan` (flagship dialog) → `/accomplish <milestone>` → repeat → `/complete <mission>`.

## Position 6 — `/complete <mission>`: close is a command, and only the operator invokes it

The close protocol gets a command — `/complete <mission>` — and one hard rule distinguishes this tier from every tier below it: **the agent never assumes, proposes-as-pending, or auto-initiates mission close.** Work orders and milestones close _themselves_ against written acceptance criteria; a mission does not. A mission is an open-ended container of operator intent, and "complete" is a judgment reserved to the human — the command exists so that when the operator decides, the protocol executes rigorously, not so the system can reach for it. (Contrast deliberately: we _always_ want a closing summary on a series of work — every `/build` and `/accomplish` produces one — but we _never_ want the system concluding the mission is done.)

Corollaries:

- **A mission may complete with open or deferred work orders** — especially a failed one. Completion is not exhaustion of the queue.
- **The archive is a snapshot**: the last state of the mission's work docs, as they stand, plus the closing report — not a cleaned-up rewrite. Open todos are triaged during the protocol (carried forward into the next mission or archived with cause); deferred/rejected proposals archive in place.
- `/complete` runs the close sequence: closing report drafted _with_ the operator (the DX retrospective is a dialog, not a form) → footgun/todo/proposal triage → `git mv` into `work/missions/<v>/` → ledger row → tag → next-mission seeding handed off to its chartering.

## Position 7 — Two closing modes; the open milestone

Positions 4–6 imply a general principle worth naming: every tier closes in one of exactly two modes.

- **Criteria-closed** — work orders, phases, campaign milestones: written acceptance, self-closing. This is the automation story, and it demands the stringent up-front specification v0.5 exemplifies.
- **Judgment-closed** — missions (Position 6), and now the **open milestone**: closed only by explicit operator decision.

An **open milestone** (`mode: open` in its frontmatter; e.g. `refine-dashboard`) is an identified _series_ whose objectives are discovered through iteration: a **direction statement** in place of fixed acceptance; no pre-filed phases; an **iteration ledger** accreting the work orders and punch lists it spawned (each of those remains criteria-closed individually — the flexibility lives at the container, not in the work); status `open`, legitimately concurrent with campaign milestones (the parallel-tracks convention already sanctions simultaneous milestones). It inherits mission-style closing semantics one tier down: operator-invoked, snapshot archive, brief closing note.

Guardrails that keep the mode from eroding the discipline:

- **Never ambiguous** — a milestone is either specified to acceptance (`mode: campaign`, the default) or explicitly declared `open`; there is no vague middle. `/accomplish` refuses an open milestone (there is nothing to converge on) — its engine is the `/build` / punch-list cadence with the operator in the loop.
- **The graduation valve** — when an iteration series stabilizes into a specifiable capability, it _spawns a campaign milestone_ rather than stretching the open one (the promotion valve pointing upward).
- **Not a dumping ground** — work orders filed against it use the existing `aligns:` frontmatter and pass normal triage; the open milestone gives the series identity and a ledger, not an exemption.
- **At mission close**, open milestones triage like open todos: close-with-note or carry forward.

This position is **adoptable mid-mission** — it's additive (a frontmatter key + one conventions line in the milestones README), requires no reorganization, and is usable in v0.5 the moment it's wanted.

## What happens now vs at the boundary

**Now (done with this proposal):** `proposals/` exists at the repo root (additive — no reorganization) and holds the workflow-design corpus; this document; nothing moves.

**At v0.5 close / v0.6 open (the migration IS the first execution of the close protocol):**

1. Write v0.5's `closing.md` (with the operator — `/complete v0.5` is this protocol's first invocation); triage open todos (forward vs archive).
2. Create `work/`; `git mv` — `docs/milestones/` → `work/missions/v0.5/milestones/`, `todos/_done/` + archived todos → `work/missions/v0.5/todos/`, mission-scoped proposals → `work/missions/v0.5/proposals/`, `docs/footguns/` → the split: v0.5-relevant files seed the next mission's triage, the `v0.4-*` archive files land as `work/missions/v0.4/footguns/`; the stray runbooks (`deploy/`, `src/services/*/deploy/RUNBOOK.md`, `docs/runbooks/`) consolidate into `work/runbooks/<slug>/` through the same triage; retro-charter v0.5 (its charter substantially exists: the milestones README preamble + M0 — formalize, don't rewrite).
3. Optionally back-fill `work/missions/v0.{1..4}/` from the `_/` stashes and archive branches, with reconstructed one-page charters/closings in the ledger (cheap, high-value: the failure lessons live there).
4. Seed the fresh `work/{mission.md,proposals,todos,milestones,footguns}` for v0.6 (footguns seeded by the Position-3 triage over v0.5's ledger); branch; update every path reference — `.claude/` agent docs and workflows, `CLAUDE.md`, `todos/README.md` (moves with it), the footgun index. **This reference sweep is the real cost of the rename and the reason it only happens at a boundary** — file it as a work order in the close protocol, not a side effect.

## Interactions with filed work orders

- `todos/26-07-02.git-workflow.md` **rule 6 (archives) is revised by Position 2**: tags-and-delete narrows to truly-dead material; work-doc archives go to `work/missions/<v>/`. The work order carries a note to align with whatever this proposal has settled to by execution time.
- The introspection and git-workflow orders are unaffected otherwise; the runner/queue design in the companion proposal gains a cleaner scan root (`work/todos/`) when the time comes.

## Open questions

1. **Standing vs mission-scoped proposals.** Some proposals outlive missions (this one; the agency-workflow design). Lean: `work/proposals/` persists across missions and only _adopted-and-consumed_ or dead proposals archive into the closing mission; a `mission:` frontmatter key marks the scoped ones.
2. **Does `docs/objectives/` fold in?** It's proto-proposal / proto-charter material. Lean: fold into proposals (pre-decision) or the relevant mission's charter (decided) at the boundary; retire the directory.
3. **Mission naming.** Version numbers (`v0.6`) vs named missions (`time-and-rhythm`)? Lean: `v<version>.<slug>` directories — the number orders, the slug carries meaning, matching the milestone convention (`3.music-nrt`).
4. **Where does the deictic/inference mission live** — a mission in this repo or (more likely) the founding mission of the agency-tooling repo, chartered under this protocol from day one? Deferred to that mission's own chartering.
5. **The campaign logs** (`todos/_done/*.accomplish-*.md`) are the richest single documents in the corpus — do they deserve a distinguished place in the archive (e.g., `work/missions/<v>/campaigns/`) or stay among the todos? Lean: stay put; the closing report's reader's index points at the important ones.
