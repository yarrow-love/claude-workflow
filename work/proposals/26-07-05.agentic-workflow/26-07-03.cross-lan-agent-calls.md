# Cross-LAN agent calls — routing consults to the machine that owns the session

> Proposal, 26-07-03. Operator insight that seeded it: "consider how the machines can call across the LAN. The machines are already networked; we would just need SSH access (basic) or an agent-service API, so that running agents can call agents on the other machine back. In the future, Tailscale or a relay could extend this beyond the local studio."

## The concept

This realizes the **machine dimension** of the callback registry that [`planner-callback`](../_enacted/26-07-05.agentic-workflow/26-07-03.planner-callback.md) explicitly deferred: "the agency layer's per-machine runner eventually routes consults to the owning machine — that's the precise agent intercom generalized." Session transcripts are **machine-local** (`~/.claude` on the host that ran the session), so a `--resume` of a session owned by another box **must execute on that box**. The `<machine>` in the [`agent-callback-identity`](../_enacted/26-07-05.agentic-workflow/26-07-03.agent-callback-identity.md) grammar (`<role>@<machine>:<session>`) is precisely the routing key: to consult `milestone-planner@bench:…` from `play`, dispatch the resume _on `bench`_ and stream the answer back.

Concretely, the studio today: plan on one box, build on another, and a phase Orchestrator that hits a plan-interpretation fork can consult the planner that authored it — even though that planner's transcript lives on the other machine.

## The design forks

**Transport — SSH-exec vs agent-service API.**

- **SSH first (basic, zero-infra, matches "SSH access now").** `ssh <machine> 'claude -p --resume <session> --model fable "<consult prompt>"'` — the answer returns on stdout, synchronously, exactly like a local consult. No daemon, no ports, no new service to keep alive. LAN + SSH keys is the whole trust model.
- **Agent-service API (later, when the runner framework lands).** A small per-machine daemon exposing dispatch/consult over HTTP — structured requests, auth tokens, streaming (`stream-json` relayed), queueing, health. This is infrastructure; it belongs with the deferred agency layer, not now.

Lean: **SSH now, API when the per-machine runner exists.** The SSH form is a documented recipe + an allowlist entry, not a build.

**Result return.** Synchronous stdout (SSH) is sufficient for consults — the caller blocks on a bounded question and distills the answer into its work doc, same as a local consult. Asynchronous callback/poll is an API-era concern.

**Cross-machine session threading.** A cross-machine call adds its own entry to the chain: the remote resume mints a new session on the remote box, recorded as `<role>@<remote-machine>:<new-session>` in the artifact's `agent:` list. The list stays a faithful, machine-qualified lineage across hosts — dial any ancestor, and its `<machine>` says where to dial.

**Auth / trust / reach.** LAN + SSH keys in the studio now. **Tailscale or a relay** (operator's future note) extends the same `<machine>`-routing beyond the studio without changing the grammar — the machine name resolves through the overlay instead of the LAN. Nothing above the transport layer changes.

## Boundary and non-goals

- This is the "larger framework" that planner-callback _and_ [`agency-workflow-design`](_deferred/26-07-02.agency-workflow-design.md) deliberately deferred; this proposal **captures the design, not a build**.
- It intersects the [`git-workflow`](../docs/git-workflow.md) doc's runner claim-protocol future note (a remote runner claims work docs via `claimed-by:` + push-conflict-as-lock) — the same per-machine-runner idea, seen from the git side. The two should co-adopt.
- **Not** a general distributed orchestrator. Studio-LAN scale first; a handful of trusted boxes; consults and dispatches, not a cluster scheduler.

## Reconciliation notes (26-07-03, full-context review — mechanics the SSH rung must handle)

1. **`--resume` resolves sessions per project directory.** Transcripts live under `~/.claude/projects/<cwd-slug>/` — the remote invocation must run with the repo as its cwd on the remote host: `ssh <machine> 'cd <repo-path-on-that-machine> && claude -p --resume …'`. Repo paths differ per machine, and the Windows workhorse's path **contains a space** (the M3.5 `MUSIC_WORKER_CMD` footgun's sibling) — quote accordingly; a per-machine `repo-path` column belongs in the same small machine registry the identity proposal wants.
2. **Non-interactive SSH shells carry a reduced PATH** — bare `claude` may not resolve (the exact failure class as M3.5's bare-`uv`: a shell started before/without the login profile lacks it). The recipe should use an absolute path or a login shell (`ssh <machine> 'bash -lc "…"'`); Windows-side, the default OpenSSH shell (cmd vs powershell) changes the quoting — settle it once in the registry.
3. **The allowlist entry must exist on the _calling_ machine's project settings** (`Bash(ssh:*)` scoped as tightly as the syntax allows), mirroring the `Bash(claude:*)` lesson: without it, consults from `acceptEdits` children auto-deny silently.

## Depends on / enables

- **Depends on** [`agent-callback-identity`](../_enacted/26-07-05.agentic-workflow/26-07-03.agent-callback-identity.md) for the `<machine>` routing key. Without machine-qualified callbacks, there is nothing to route.
- **Enables** the cross-machine half of the whole call hierarchy — the reason the operator can already run the system all night on one box, extended to the whole studio.

## Spawn plan (when adopted)

A `todos/` work order for the SSH rung only: a documented consult-over-SSH recipe in the executor/orchestrator escalation ladder (resume-on-owning-machine when the callback's `<machine>` ≠ local), an SSH-key setup note, and a `Bash(ssh:*)` allowlist entry scoped to studio hosts. The service-API rung defers to the per-machine runner framework.
