# Planner callback — consulting the planning session instead of spawning fresh agents

> Proposal, 26-07-03. Operator insight that seeded it: after `/plan docs/milestones/5.timeline`, a Fable session exists with excellent milestone context — "it would be ideal if the orchestrator could call the same Fable planning agent back with questions rather than calling a fresh agent." This proposes exactly that, using mechanisms that already exist; no larger agentic framework required.

## The enabling fact: resume is a fork

`claude -p --resume <session_id>` does not continue-and-mutate the original session — it **forks** it: the resumed session begins with the complete context of the original and receives its own new session id. Consequences:

- A completed `/plan` session is a **frozen, forkable instance of the planner at maximum context** — the milestone references read, the forks argued, the operator's answers absorbed.
- Multiple consumers (several phase Orchestrators, the Executor) can consult it **in parallel**, each in their own fork; the canonical planning transcript is never corrupted.
- The introspection upgrade (landed 26-07-03) already made session-id capture at launch a standing habit — this proposal extends the same discipline one tier up.

The same principle exists **intra-session** already: the Agent tool supports continuing a previously-spawned subagent with its context intact (SendMessage by agent id/name). An orchestrator that dispatched a planner subagent can call _that same planner_ back mid-campaign — today, with zero new mechanism.

## The design — a consultation channel, not a chat channel

**Rung 0 — intra-session callback (adoptable immediately).** Orchestrator guidance: when a question arises about a plan _during the campaign that executes it_ (a reviewer finding that seems to contradict a plan decision, an implementer discovering the plan's assumption doesn't hold), prefer **calling back the planner subagent that wrote the plan** over dispatching a fresh researcher/planner — the author already holds the rationale. One line in `orchestrator/CLAUDE.md` delegation guidance.

**Rung 1 — cross-session callback (small, adoptable when wanted).**

1. `/plan docs/milestones/<m>` records its own session id at close — `plan-session: <id>` in the milestone `index.md` frontmatter (alongside the machine that ran it; see Limits).
2. Orchestrators/Executors gain a **consult step** in the escalation ladder: when a genuine question arises that the phase docs don't answer, _before_ returning `ESCALATE` to the human —

   ```sh
   claude -p --resume <plan-session-id> --model fable \
     "Consultation from the M5.2 build (work doc: <path>). Question: <the specific fork>.
      Context that changed since planning: <delta>. Answer from the plan's intent; flag if this needs a human decision."
   ```

3. The answer is **distilled into the work doc** (the work doc stays canonical; consultations are advisory input, never a side-channel of record).

**The boundary that keeps it honest — interpretation vs decision.** The resumed planner answers _interpretation_ questions: what did the plan intend, which reading of the acceptance criterion is right, why was X rejected. It does **not** own _new_ design decisions — those still escalate to the human, exactly as today. The consult prompt says so explicitly ("flag if this needs a human decision"), and the orchestrator treats "this is a new decision" answers as ESCALATE, not as license. Without this line, the callback channel would quietly erode the human sign-off gate.

**Staleness is a feature, correctly framed.** The planner's context is frozen at planning time; the tree moves during the campaign. That is fine — you are consulting **the author about intent**, not an oracle about current state. The consult prompt carries the delta ("context that changed"); anything requiring fresh code truth goes to a researcher as usual. The two are complements, not substitutes.

## Cost shape

Each consult forks the full planning transcript at Fable input rates — real but bounded, and precisely targeted: it spends flagship tokens only at the moments a campaign hits a genuine fork (the escalate-on-novelty path, historically rare). Compare the alternative: a fresh agent rebuilding milestone context from documents costs comparable input _and_ loses the unrecorded rationale. Reserve consults for the escalation path, never routine questions; the `## Progress` ledger + work docs answer the routine ones.

## Limits (and where the future framework picks up)

- **Machine-local.** Session transcripts live in the machine's `~/.claude` store — a consult only works on the machine that ran `/plan`. Record `plan-machine:` beside `plan-session:`; in the studio today (plan and build usually on the same box) this rarely bites. The agency layer's per-machine runner eventually routes consults to the owning machine — that's the "precise agent intercom" generalized, and it's where this proposal hands off to the larger framework deliberately not pursued yet.
- **Retention.** Old transcripts can be pruned by the CLI; a mission-length gap between plan and consult may find the session gone. Degrade gracefully: consult-if-present, fall back to documents + fresh dispatch.
- **The general form** (deferred with the framework): a role→session-lineage registry — every long-lived context (mission charter dialog, milestone plan, campaign executor) registered and consultable by role name rather than raw id. This proposal is that idea at n=1, proven on the highest-value context first.
