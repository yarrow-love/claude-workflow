# Per-section provenance — stamp every appended work-order section with its author's callback id

> Proposal, 26-07-05. Operator insight that seeded it, observed while watching a live build: the `todos/<work-order>.md` is **running durable state** that the orchestrator and every specialist write to, one section per operation (implementation, review, resolution, inspection, …). The frontmatter already carries the document's session lineage as a `sessions:` list (`<role>@<machine>/<session-id>`). The proposed upgrade: **preface each appended section of the body with the same `<role>@<machine>/<session-id>` stamp for the agent that wrote _that section_** — so the work order is not merely a historical record but a set of inline callback anchors, each of which can be dialed back with `claude --resume <session-id>` to introspect the authoring agent's reasoning.

## The concept

Today the work order accumulates sections as the build progresses — an implementer appends what it built, a reviewer appends findings, a resolver appends how it fixed them, an inspector appends what it verified. The reader can see _what_ each operation produced, but not _who_ produced it or _how to ask them why_. The `sessions:` frontmatter list ([session-callback-ids](../../.claude/projects/-home-yarrow-dev-lasers/memory/session-callback-ids.md), LOCKED 26-07-05) records the aggregate set of sessions that touched the document, but it is a flat bag: it does not bind a given _section_ to the session that authored it.

This proposal closes that gap. Each appended section is prefaced by the writing agent's callback id, e.g.:

```markdown
## Implementation — service band-energy channel

> executor@play/1063f991-16dd-4fef-bc25-ba8c37c9dd2c

Added the `service:band-energy` channel to … (section body follows)

## Review

> reviewer@play/86ade408-7a9b-4edd-991a-6b3592ce377a

Two findings: … (section body follows)
```

Now the document is **doubly useful**: it remains the durable state the workflow reads and writes, and it becomes a directory of **live callback handles** keyed to the exact work each session performed. An agent (or the operator) reading a suspicious review finding can resume `reviewer@play/86ade408-…` and ask "why did you flag this — what did you consider?"; the reasoning that produced the section is recoverable at full context, not lost to the transcript.

## Why this is distinct from the frontmatter lineage

[`agent-callback-identity`](26-07-03.agent-callback-identity.md) proposes an `agent:` list stamping the **call chain that _originated_ the document** (mission-planner → milestone-planner → executor → …). That is provenance for the document's _creation lineage_. This proposal is orthogonal and complementary: it stamps the **per-operation authorship of the document's _growth_** — every subsequent write, by whichever specialist performed it, not just the originating ladder.

- Frontmatter `sessions:` / `agent:` — _who brought this document into being and who has touched it_ (aggregate, at the top).
- Inline section stamps — _who wrote this specific paragraph, and the exact session to ask about it_ (granular, at the point of the work).

Both use the same grammar (`<role>@<machine>/<session-id>`), so adopting this costs no new vocabulary. The inline stamp can also be treated as the **canonical source** from which a `sessions:` frontmatter list is trivially derived (dedupe the section stamps) — one convention feeding the other rather than two hand-maintained lists drifting apart.

## Grammar

Reuse the locked form verbatim — [session-callback-ids](../../.claude/projects/-home-yarrow-dev-lasers/memory/session-callback-ids.md):

```
<role>@<machine>/<session-id>
```

Rendered as a blockquote line immediately under each section heading (`> <role>@<machine>/<session-id>`), so it is visually subordinate to the heading, greppable (`grep '^> .*@.*/' work-order.md`), and machine-parseable. A section that spans multiple operations by the same session needs only one stamp; a section revised by a _different_ session appends a second stamp line rather than overwriting the first (the revision chain stays visible).

## Why it matters — from record to introspection surface

The operator's framing: "we have more than an historical record — we have callback identifiers that enable either agents or the operator to introspect into agents' thinking process by calling back the agent session that performed the work with questions." The value compounds in the automatic workflows, where no human is in the loop during the build: when a later phase distrusts an earlier decision, it can _consult the source_ instead of re-deriving or guessing. The work order becomes the index into every mind that shaped it.

## The boundary that keeps it honest

Consistent with the sibling proposals: a callback is **provenance plus a consult handle, not authority**. Resuming a section's authoring session answers "what did you intend / what did you consider / why did you rule X out" — it does not delegate _new_ decisions to that session. The stamp makes each section's author _reachable_; it does not make any past session _the boss_ of the current one.

## Open forks (for the adopting work order to settle)

- **Placement grammar** — blockquote line under the heading (this proposal's lean), vs. an HTML comment (`<!-- executor@play/… -->`, invisible in rendered Markdown but greppable), vs. a trailing attribution footer per section. Lean: visible blockquote — the provenance _should_ be legible to a human skimming the doc.
- **Specialist session ids** — durable resumable tiers (executor, orchestrator, planners) have `claude -p` session ids; a _transient subagent_ (a reviewer dispatched intra-session) may only have an intra-session id, not a resume target. Resolve consistently with `agent-callback-identity`'s open fork: either the subagent stamps its parent orchestrator's session (the resumable handle that _can_ answer for it), or the workflow promotes reviewers/inspectors to their own `claude -p` sessions so the stamp is directly resumable. Lean: stamp the nearest resumable session, note the subagent role in the `<role>` field.
- **Who writes the stamp** — each agent self-stamps as its close-step when it appends its section (symmetric with the `sessions:` frontmatter close-step), vs. the orchestrator stamps on the agent's behalf when it splices the section in. Lean: self-stamp, matching how each tier already records its own frontmatter entry.
- **Derivation of `sessions:`** — auto-derive the frontmatter list from the inline stamps (single source of truth), vs. maintain both. Lean: derive.

## Relationship to the sibling proposals

- Builds directly on [session-callback-ids](../../.claude/projects/-home-yarrow-dev-lasers/memory/session-callback-ids.md) (the locked `<role>@<machine>/<session-id>` grammar and `sessions:` frontmatter) — this is that grammar pushed _into the body_, per-section.
- Complements [`agent-callback-identity`](26-07-03.agent-callback-identity.md) (originating-lineage frontmatter) — orthogonal axis: creation lineage vs. per-operation authorship.
- Gated on [`cross-lan-agent-calls`](26-07-03.cross-lan-agent-calls.md) only for the _cross-machine_ consult path; single-machine introspection (resume a `@play` section stamp on `play`) is adoptable alone.
- Feeds [`accomplish-live-feed`](26-07-03.accomplish-live-feed.md) and [`planner-callback`](26-07-03.planner-callback.md) — a section-attributed work order is a richer live feed and a finer-grained callback registry than a flat session list.

## Spawn plan (when adopted)

A `todos/` work order: (1) settle the placement grammar and the specialist-session fork above; (2) add a self-stamp close-step to the implementer / reviewer / resolver / inspector / orchestrator workflows — "when you append your section, preface it with `> <role>@<machine>/<session-id>`"; (3) optionally add a derive-`sessions:`-from-body step so the frontmatter list stays in sync automatically. Additive to the existing `sessions:` convention — nothing migrates mid-flight; in-progress work orders gain stamps on their _next_ appended section.
