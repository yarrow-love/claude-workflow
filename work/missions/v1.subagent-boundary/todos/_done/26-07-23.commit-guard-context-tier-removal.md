# commit-guard context tier re-fires every stop — REMOVE it (re-scoped from "quiet")

> **Re-scoped 26-07-23 (quiet → remove).** Filed as "quiet the context tier to once-per-set" and blessed as such (both preserved below). Then the defect recurred against a *legitimately-paused* session and the Architect, re-consulted, ruled the whole tier a design error to remove — not quiet. The current live plan is **§ Removal Plan**; the § Refactor Plan and its § Blessing below are the SUPERSEDED quieting approach, kept as the decision trail.

## Refactor Objective

**What**: `.claude/bin/commit-guard` (the follow-through Stop hook, added 26-07-23 by `work/todos/_done/26-07-23.commit-follow-through.md`) emits its **context tier** reminder (non-blocking `additionalContext`) on *every* Stop while any `work/`/`.claude/`/`docs/` path is dirty. The block tier is loop-guarded (`stop_hook_active` + a `tmp/` marker keyed by session + block-path-set fingerprint); the **context tier has no such guard**, so it repeats verbatim each turn.

**Why**: observed live while filing this repo's own commit-follow-through order — an interactive manager session idle-waiting on background subagents re-triggered the context reminder on every idle turn (~10× in one session). The ambient-noise residual was explicitly accepted by the Architect **for the mid-build case**, but "re-fires every turn while idle-blocked" is sharper than "ambient awareness." It reads as a busy-loop, and the repetition dilutes the signal.

**Proposed fix**: extend the existing marker mechanism to the context tier — emit the context reminder at most once per `(session_id + fingerprint(sorted context-path-set))`. Re-arm when the dirty set changes (a new artifact appears). Keeps ambient awareness on first notice; kills the idle repeat. Reuse the `tmp/commit-guard/` marker dir and the same SHA-256 fingerprint helper already in the script; distinguish context markers from block markers (e.g. a `ctx.` prefix) so they don't collide.

**Constraints**: preserve every current behavior of the block tier (unchanged); keep the "buggy backstop never breaks a session" property (any error → exit 0 silent); genericity (nothing project-specific); stale-marker accumulation in `tmp/` remains the separate deferred cleanup noted in the parent order's Out of Scope.

**Acceptance**:
- A stable dirty tree reminds once per session per unchanged context-path-set, not every stop; a changed context-path-set re-arms and reminds again.
- Block-tier behavior byte-for-byte unchanged; all 8 cases of the parent order's Phase D demonstration recipe still pass.
- Any internal error still exits 0 silently.

3. **Fold the ctx-marker guard into `emitContext`**, before the existing emit: fingerprint the `paths` array **verbatim** (`sha256(paths.join("\n")).slice(0,16)` — inputs are already sorted+deduped and the two halves are disjoint by construction, so no re-sort); build `join(markerDir, ` + "`${sessionId}.ctx.${fingerprint}`" + `)`; if it exists → `return` (caller's `exit(0)` makes the stop silent); else write the marker and fall through to emit. **Wrap only the `mkdirSync`/`writeFileSync` in a local try/catch** — on write failure, still emit (over-remind is safe; a silent break is not — and this helper sits outside the outer catch, so an unguarded throw here would be uncaught). Mirrors the block tier's own marker-write guard.

The guard is needed because that outer catch would then *swallow the emission* — a marker-write failure would produce silence, when the correct degradation is to **over-remind** (write failed, still emit). (`existsSync` doesn't throw; if it somehow did, outer-catch → silent exit 0, which the filed acceptance permits.)
4. **Update the two call sites** — line 109 (`emitContext(contextPaths, …)`) and line 125 (`emitContext([...blockPaths, ...contextPaths], …)`).

**Decision points** (both settled):
1. **Silent-on-repeat** (recommended) vs. a terser second reminder — silent matches the block tier's "once per offending set" model and the filed acceptance; a second reminder just reintroduces the noise this order kills.
2. `.ctx.` marker prefix — kept for legibility/defence (collision with block markers is already structurally impossible).

**Verification** (corrected per Blessing D1): "block-tier unchanged" means the block **decision** path (fingerprint → marker → `{"decision":"block"}`) is byte-for-byte untouched — genuine close-out evidence still forces its turn.
