# `work/PROJECT.md`'s sanctioned-absence list omits `src/_archive/`

Filed 26-07-24 from an M3 phase-1 quality-gate run. Not investigated — symptoms and location only.

## Symptom

Quality-gate item 1 (reference resolution, `work/PROJECT.md:9`) reports four unresolved paths on a clean tree:

```
src/_archive/
tmp/accomplish
tmp/transcript
work/todos/BACKLOG.md
```

The last three are named in `work/PROJECT.md:9`'s sanctioned lazy-create list. **`src/_archive/` is not** — so every gate run produces a miss the operator must adjudicate from memory, and a strict reader of the gate would call it a failure.

It is in fact a sanctioned absence, and the product says so explicitly: `.claude/BASELINE.md:38` describes `src/_archive/` as "a convention path created on first retirement — **reference gates must sanction its absence**." The mandate exists in the product; the consuming-side list that implements it (`work/PROJECT.md`) never picked it up. Introduced with the reference at `fd73dc6` (codebase-memory-mcp baseline).

## Where

- `work/PROJECT.md:9` — the sanctioned lazy-create enumeration (missing entry)
- `.claude/BASELINE.md:38` — the mandate it should implement
- Also referenced: `.claude/BASELINE.md:54`, `.claude/agents/researcher/CLAUDE.md:26`, `.claude/commands/initialize.md:8`

## Why it matters beyond cosmetics

`work/PROJECT.md` is itself stripped by the M3/D4 template purification, and the M3 reconnaissance already flagged (concern C5) that the fold rehearsal's sanctioned-absence list must be written **into the runbook** rather than assumed resident. This is a fourth member of that list, discovered after C5 was written. Route to M3 phase 3 rather than fixing in isolation.

## 26-07-24 — split across phases (supersedes the routing above)

The M3 phase-2 Reconnaissance gate (confirmed decision 3) splits this bug in two:

- **The `work/PROJECT.md:9` half is fixed in M3 phase 2.** The sanctioned lazy-create enumeration now names `src/_archive/`, and — the same defect on two further paths the gate surfaced — `work/PROJECT.md` itself (authored at bootstrap step 3, so absent in a bare template tree, yet referenced by 17 files under `.claude/`) and `work/sessions/MANIFEST`. Without those, phase 3's purification rehearsal runs red against the stripped tree.
- **The runbook half remains phase 3's.** Concern C5 stands: the fold rehearsal must carry the sanctioned-absence list **written into its runbook** rather than assuming `work/PROJECT.md` resident — the file the list lives in is itself stripped by purification.

Stays `open`; phase 3 closes it once the runbook half lands.

## Resolution (26-07-24)

Verified against `work/PROJECT.md:9` — the two lists match exactly: `src/_archive/`, `tmp/*`, `work/MISSION.md`, `work/PROJECT.md`, `work/todos/BACKLOG.md`, `work/sessions/MANIFEST`, `work/design/` contents.

The rehearsal exercised it in anger: reference resolution against the purified tree extracted 30 distinct paths and reported **7 unresolved, every one covered by this list** — no red on `work/PROJECT.md` beyond its sanctioned absence, so no phase-2 escalation.

**One finding routed onward rather than absorbed here.** The same section records a divergence this bug did not name: `.claude/BASELINE.md`'s "Paths that post-date the scaffold" enumerates only **3** of the 7. A consuming repo authoring its gate from BASELINE alone goes red on `tmp/*`, `work/MISSION.md`, `work/todos/BACKLOG.md`, and `work/design/` contents. That is the mirror-image of this bug — the *product* side of the same gap — and it is filed separately as [`26-07-24.bug.baseline-postdate-list-divergence.md`](26-07-24.bug.baseline-postdate-list-divergence.md), carried forward because `BASELINE.md` is final `main`-bound text and phase 3 is barred from doctrine authoring.
