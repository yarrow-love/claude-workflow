## Feature Request

Build the transport rung of the [lan-intercom proposal](../proposals/26-07-22.lan-intercom.md) (adopted design; the proposal carries the full rationale and determinations — read it first).

**What**: a per-machine Bun consult server (`.claude/bin/`, sibling of `consult`) exposing `POST /consult` `{repo, handle, question}` and `GET /health`; the `bin/consult` machine-mismatch seam extended to resolve the handle's `<machine>` to its mDNS host via the registry and POST; the machine registry (`work/README.md`) gains a host column (`play.local`, `antec.local`, `benpc.local`); a `work/runbooks/` runbook for key distribution + LAN-only firewall setup per machine.

**Why**: makes every stamped callback consultable across the studio — the callback grammar's `<machine>` segment becomes routable, ahead of any second machine actually joining the work.

**Constraints** (determinations from the proposal — not open for re-decision at plan time):
- Shared bearer key **required**; untracked env file per machine; LAN-only firewall as second layer.
- **Repo-scoped authorization**: only registered repos, `cwd`-pinned; never a general shell.
- **Read-only forks**: every resume dispatched with a no-writes permission profile.
- **Model derived from the handle's role** via the existing pins (architect/planner → fable, manager/executor → opus, specialists → sonnet); one-tier fallback on unavailability.
- Genericity: the server is bootstrap tooling — project-generic, BASELINE-manifested, no machine facts in `.claude/` (hosts live in the registry; keys in untracked env).

**Acceptance**:
- `consult <remote-handle> "<question>"` on one machine returns the remote fork's answer end-to-end (demonstrable single-machine via loopback against `play.local`).
- Requests without the key are rejected; a consult fork provably cannot write (show the denied write).
- A repo not in the server's registry is refused.
- `GET /health` answers; `consult` fails crisply (named cause) when the target is unreachable.
- Registry host column populated; runbook present; `BASELINE.md` manifest updated in the same change.

**Out of scope** (second rung — file when a second machine joins the work): the claim protocol (`claimed-by:` + push-conflict-as-lock), `blast:` radii, stale-claim rule, and their codification in `work/todos/README.md` + manager/executor docs.

## Research Findings

Read-only grounding pass. The design is buildable on the existing surfaces; five genuine forks surfaced for the plan to resolve.

**The client seam is exact and small.** `.claude/bin/consult` parses the handle with a named-group regex (`role`/`machine`/`uuid`/`msg`; `machine` defaults to the local hostname) and the machine-mismatch branch is a single `if (machine !== machineLocal) { console.error(...); process.exit(2); }` block (`consult:51-57`). That block *is* the plug-in point: its body is replaced with registry host-lookup → `POST /consult` → stream the answer, degrading to a crisp named error (unregistered / unreachable / key-rejected) on failure. The restore-before-resume ladder (local cache → `work/sessions/<machine>/<uuid>.jsonl.gz` gunzip → exit 3) and the fable-hardcoded `--model` default (`consult:31`) stay as-is client-side — model derivation is a **new, server-side** behavior. House style to match: `#!/usr/bin/env bun`, `node:`-prefixed builtins (no `Bun.*` in the one-shot scripts), `spawnSync` with `stdio:"inherit"`, manual `process.argv` parsing, `console.error` for diagnostics, explicit exit codes (1 usage / 2 mismatch / 3 no-transcript / passthrough). The `repoRoot`/`slug`/`projectDir` boilerplate is duplicated verbatim across both existing scripts — a new server would repeat it; not this rung's job to de-dup.

**Sibling contract.** `.claude/bin/package.json` `bin` map gains a `"consult-server": "./consult-server"` entry; BASELINE tooling bullet (`BASELINE.md:23`) and its PATH prose gain the new command. The quality-gate smoke test asserts `bun .claude/bin/<script>` with no args prints usage and exits 1 — the server must honor an equivalent crisp-fail contract (e.g. `--help` prints usage; missing bearer-key env fails named). `Bun.serve({ port, fetch })` is genuinely new ground — no server code exists in `bin/` to imitate; the argv/diagnostic idioms still bind.

**Registry: new rows, not just a column.** Current table (`work/README.md:95-100`) has four columns — `Machine | OS / shell | Repo path | Notes` — and **only the `play` row exists**. Acceptance's "host column populated" therefore means adding a `Host` column *and* the `antec`/`benpc` rows (`antec.local`, `benpc.local`). The callback grammar already declares `<machine>` the "routing key for any future cross-machine consult" (`work/README.md:79-85`) — no grammar change needed.

**Scaffold gaps this rung must absorb.** (1) `work/runbooks/` does not exist despite being BASELINE-mandated scaffold (`BASELINE.md:38`, `work/README.md:18`) — the plan creates it with a README (first runbook the repo holds; no prior example to imitate). (2) `.gitignore` has **no `.env`-shaped pattern** — the untracked bearer-key file (e.g. `.claude/bin/.consult-server.env`) needs a tracked, generic ignore pattern added in the same commit (pattern generic; value machine-local — satisfies the genericity constraint). BASELINE manifests only `.claude/` paths, so `work/runbooks/` needs no manifest entry; the `.claude/bin/` additions do.

**⚠ FORK 1 — the read-only guarantee is the security crux and has no single clean flag.** Claude Code exposes no `--permission-mode readonly`. The robust boundary is an explicit deny-list — `--disallowedTools "Edit,Write,Bash(*),NotebookEdit"` or the equivalent `--settings` inline `{"permissions":{"deny":[…]}}`. `--permission-mode default` is *not* a reliable read-only guarantee: headless `-p` blocks writes largely as a side effect of having no TTY to prompt, not by semantic guarantee. **Treat `--disallowedTools`/`permissions.deny` as the boundary, `--permission-mode` as defense-in-depth only.** Acceptance's "show the denied write" forces an empirical probe: a consult that asks the fork to edit a file, capturing `--output-format json` proving the edit was denied and the process still answered (exit 0). This probe must run before the rung is trusted.

**⚠ FORK 2 — process supervision unspecified.** Neither proposal nor order says how the Bun server stays alive (systemd unit / manual foreground / launchd). A runbook decision — planner to scope: codify one supervision recipe vs. leave to operator discretion.

**⚠ FORK 3 — server "no-args" semantics.** A long-running server has no positional args; the planner must define what the crisp-fail contract means for it (missing key env → named exit; `--health`/`--help` affordance) to stay smoke-test-consistent.

**⚠ FORK 4 — role→model mapping precision.** The order restates `manager`/`executor` → opus, but the authoritative pin table (`manager/CLAUDE.md:86`) pins `implementer`/`designer` → opus and treats `planner` → fable, rest → sonnet; `manager`/`executor` aren't dispatched-and-pinned specialists. The planner must fix the exact `<role> → <model>` lookup the server uses against the real pin source, not the proposal's paraphrase, with the one-tier-down fallback.

**⚠ FORK 5 — new exit code / error taxonomy** for the extended client: "unreachable target" and "key rejected" are new failure modes that shouldn't reuse code 2 (mismatch) or 3 (no-transcript). Planner to assign the taxonomy.

**Non-issues confirmed.** `<machine>.local` resolves through the OS resolver — Bun `fetch` needs nothing special (mDNS install is a runbook precondition). `Bash(claude:*)` is already baseline-allowed (covers the server's `claude` dispatch); no new permission entry needed for the server's operation.

### Decision points

1. **Read-only boundary** — three layers composed: inline `--settings permissions.deny` (`Edit,Write,NotebookEdit,Bash(*),WebFetch,WebSearch`) + `--disallowedTools` + `--permission-mode dontAsk`, **plus MCP severance** (`--strict-mcp-config` + empty `--mcp-config` → the fork loads zero MCP servers) *(recommended)* vs. `--permission-mode` alone or deny-list-only. Deny rules bind even under the target repo's `bypassPermissions` (this repo sets exactly that); mode-alone is defeatable; deny-without-severance leaves user-scope Sanity/Higgsfield/Playwright-class MCP tools live (a deny list can't enumerate them — severance is the only allowlist-shaped posture). Cost: the fork loses read-only codebase-memory MCP. Consequence: the security determination actually holds.
2. **`--fork-session` on both invocations + correct `work/README.md:85`** *(recommended)* vs. trust `--resume` to fork. Current docs say `-p --resume` mutates in place; omitting the flag corrupts the never-mutate-the-transcript promise. (See ESCALATE 2.)
3. **Buffered response** *(recommended)* vs. chunked streaming (proposal says "stream the answer"). Buffering buys the model fallback + clean 502; cost is no incremental output on long consults.
4. **Role→model as a server-side constant table** *(recommended)* vs. parsing agent frontmatter at runtime — the durable tiers (manager/executor) have no pins to parse; table carries a citation comment. manager/executor→opus is a served default, flagged as such.
5. **Supervision codified (systemd user unit) in the runbook** *(recommended)* vs. operator discretion. The runbook exists to make machine setup repeatable; an unsupervised server dying silently defeats "the grammar becomes routable."
6. **Registry home = `work/PROJECT.md` seam** *(operator-directed)* — the concrete machine table lives in PROJECT.md (the per-project seam), populated at machine-join via the runbook; `work/README.md` keeps only the grammar, schema, and registration convention. Vs. keeping the live table (or a one-row example) in README.md — rejected: an example row alongside a PROJECT.md live table is a drift engine, and assumed rows in the generic-copied README violate the two-hats rule.
7. **Registry read = header-indexed markdown parse (of PROJECT.md)** *(recommended)* vs. a `work/machines.json` sidecar (rejected — second source of truth / drift).
8. **Server no-args = explicit `serve` subcommand** *(recommended)* vs. bare-starts-server (would hang the smoke test on configured machines).
9. **Exit taxonomy 4/5/6/7, reuse 3, retire 2** *(recommended)* vs. reassigning 2 (rejected — historic docs reference its old meaning).
10. **`/health` unauthenticated** *(recommended)* vs. keyed (breaks curl diagnostics; firewall is the second layer; leaks only liveness+version).
11. **Default port 9999, two-way configurable** *(operator-directed)* — server side via `CONSULT_SERVER_PORT`/`--port`; client resolves the target **per-machine from the registry Host cell** (`host:port`), bare host → 9999. A machine that can't bind 9999 serves elsewhere and advertises it in its own row.
