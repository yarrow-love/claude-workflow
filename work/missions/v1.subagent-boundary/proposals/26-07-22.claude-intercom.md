# claude-intercom — a per-machine, multi-repo agent service with a front-end

> Deferred proposal, 26-07-22, seeded from the operator dialog that adopted [`lan-intercom`](../26-07-22.lan-intercom.md). Operator framing: "a per-machine, multi-repo/project (claude-intercom) cross-platform (Linux, Windows, Mac) application, which we could provide a front-end for … a single service running per machine, where operators would be able to collaborate on multiple repos simultaneously." Explicitly "an experimental nice-to-have and not an immediately actionable request" — this document records the why and the how so the context survives until the triggers fire.

## Why this is the named terminus, not a new idea

Three deferred designs are fragments of this one service: the per-machine runner / agent-service API rung ([`cross-lan-agent-calls`](../26-07-05.agentic-workflow/26-07-03.cross-lan-agent-calls.md)), the work-queue tier's dispatch endpoint ([`work-queue-tier`](26-07-11.work-queue-tier.md)), and the lan-intercom consult server (in build). Naming the terminus prevents drift: each rung is built knowing what it grows into, and — critically — knowing what it must *not* absorb early.

**The differentiator is the work system, not the transport.** The ecosystem has HTTP control planes over coding agents (coder/agentapi), an interop standard (A2A v1.0, Linux Foundation 2026; claude-a2a wraps Claude Code as an A2A agent), and same-machine orchestration (Claude Code Agent Teams, parallel-sessions UI, many orchestration UIs). None have repo-scoped authorization, provenance-anchored consults (the callback grammar), or git-native coordination (claims, blast radii, stigmergic state). The transport is commodity; the work-system integration is the product.

## The rung ladder (each rung ships value alone; graduation triggers named)

1. **Transport rung** — `.claude/bin/consult-server` (the in-flight `todos/26-07-22.lan-intercom.md` order). Already multi-repo-shaped: `POST /consult {repo, …}` + registered-repos authorization. No rework needed for anything below.
2. **Runner rung** — claim/dispatch endpoint, adopted with the work-queue tier. Graduates to `src/` in this repo per lan-intercom's existing clause. Candidate substrate to evaluate before hand-rolling session control: coder/agentapi. Server-side blast-radius computation (via the machine's codebase-memory instance) is a natural feature here.
3. **Extraction rung — the fresh `claude-intercom` repo.** Trigger: the service becomes a per-machine *installed product* — multi-repo config, own release cycle, cross-platform packaging (Linux/Windows/Mac). Integration back into consuming repos follows the **codebase-memory pattern** (the established precedent for machine-level services): installed binary/service, BASELINE expectation, `/initialize` audits reachability, thin `consult` client remains in the bootstrap. The new repo bootstraps with this workflow — dogfood instance #2.
4. **Front-end rung.** Lean: **browser-first** — the server speaks HTTP; a localhost-served `/ui` dashboard (flow diagrams of campaigns, live feeds — the accomplish-live-feed lineage; the work-queue tier's eventual control room) costs no shell and is cross-platform free. A native shell (Electrobun keeps the single Bun runtime but is young; Wails/Tauri are mature but add a language) only if OS integration (tray, notifications) proves necessary.

## Design notes recorded for activation time

- **Protocol lean: watch A2A.** Speaking A2A v1.0 instead of a bespoke protocol would buy interop with the standardized agent ecosystem. Decide at the runner rung.
- **Auth evolves with collaborators**: membership is governed by the invite-only git repo, so the machine-global bearer key (transport rung) should evolve to **per-repo keys** — the auth boundary aligned with the collaboration boundary. Each repo carries its own network of collaborators; the service multiplexes.
- **codebase-memory: single per-machine instance, already the design.** One binary, one store, projects keyed by `repo_path`, `CROSS_*` edges across repos in one store. Nothing to fold; claude-intercom mirrors this shape and consumes the same instance.
- **Shrink risk, accepted**: if Anthropic ships cross-machine Agent Teams, the transport/runner core shrinks to the repo-scoped-security + work-system layer. That layer is the point; re-evaluate scope at each rung against the then-current ecosystem.

## Activation triggers

Any of: (a) a second machine is *actually* working claimed orders in parallel and the transport rung has proven itself; (b) a genuine multi-repo need (two projects consulting across the same studio); (c) the work-queue tier's adoption reaches for its dispatch endpoint; (d) a collaborator beyond the operator joins a repo's network. Until then this stays deferred — the transport rung is deliberately dumb, and nothing in it should anticipate this document beyond what its proposal already carries.
