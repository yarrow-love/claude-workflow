# Subagent Manager boundary — native observability for the Executor→Manager tier

> Proposal, 26-07-23, from an operator dialog. Seeding insight: the Manager-as-headless-child pattern rests on "an incorrect assumption that Claude Code dispatch was still limited to ~2x subagent depth … the issue with that is that we lose observability." Confirmation condition, met and operator-blessed: "As long as the revised architecture will continue to support this call-and-response up the chain of command and back down to the awaiting inferior agent … then I confirm the architectural revision, if this buys us greater observability into the development process at every level of the hierarchical agent tree." Companion observation folded in below: the sessions archive, "implemented but not truly leveraged pragmatically," may gain its real role through Orchestra — "asynchronous observability to peer developer machines."

## The concept

Convert the Executor→Manager boundary from headless `claude -p` child sessions to **Agent-tool subagent dispatch**, gated on explicitly configured nesting depth. The compensatory observability apparatus (stream-json plumbing, `tmp/accomplish` jsonl logs, `tail | jq` recipes, the mtime-stall rule) retires; the operator gets the native subagent panel — live transcripts at every level of the tree, task status, surfaced permission prompts — and the chain of command gains a *stronger* escalation shape than it has today.

Everything else in the dispatch architecture survives: consult forks (cross-session, depth-irrelevant), the work order as state machine, the `## Progress` ledger (crash-safe state, not observability plumbing), the Blessing gate, archive-on-stamp.

## The verified premise (claude-code-guide verification, 26-07-22)

1. **Depth is configuration, not default.** Nesting to 5 levels was the *default* only in v2.1.172–v2.1.216; current CLI (v2.1.217+) ships with subagent nesting **off** — enabled via `CLAUDE_CODE_MAX_SUBAGENT_SPAWN_DEPTH` (`2` covers Executor → Manager → specialists). The executor doc's original hedge — "depth-limited **and version-dependent**" — has been vindicated twice in a year; the standing lesson is that this capability must be **configured and audited, never assumed**.
2. **Observability inverts.** Operator: live subagent transcripts in the native panel, task-list status, permission prompts surfaced to the main session. Parent session: final result only (no mid-flight stream) — acceptable because mid-flight state already rides the work doc's `## Progress` ledger.
3. **Continuation is native.** SendMessage works across nesting levels; a resumed subagent retains full history and picks up exactly where it stopped.
4. **Durability is parent-scoped.** Subagent transcripts live at `~/.claude/projects/<slug>/<sessionId>/subagents/agent-<agentId>.jsonl` — on disk, surviving restarts, but resumable only *through the parent session* (`claude --resume <parent> …`), never independently.
5. **Permissions: parent mode dominates.** A subagent cannot run stricter than its parent's mode.
6. **Concurrency**: ~20 concurrent subagents default (configurable), 200/session — ample at studio scale.

## The design

**1. The env gate.** `.claude/settings.json` gains `env: { CLAUDE_CODE_MAX_SUBAGENT_SPAWN_DEPTH: "2" }` as a baseline key; BASELINE's settings-skeleton section records it; `/initialize` audits both presence and (post-spike) a behavioral check. Depth `2` exactly — the ladder needs no more, and a tight cap is the blast-radius principle applied to dispatch.

**2. Dispatch conversion** (`execute-milestone.md` 7.1 + `executor/CLAUDE.md` § The Manager boundary). The Executor dispatches each phase Manager as an Agent-tool subagent — per-call `model: "opus"`, the phase work doc as the self-contained brief, the same automatic-mode workflow doc governing it. Retired: the `--output-format stream-json` invocation, `tmp/accomplish/` logs, the 20KB-tail check-in recipe, the mtime-stall rule (replaced by task notifications + the panel), session-id capture from the init event (replaced by the returned agent id). The bounded check-in habit survives in spirit: the Executor reads the `## Progress` ledger, not a transcript.

**3. Call-and-response, both directions** (the operator's confirmation condition — this is the protocol, stated once):
- **Upward: pause-and-ask.** A Manager that hits novelty ends its turn with the question — context held, truly *awaiting*. The Executor deliberates (consulting the Architect per the existing ladder — cross-session fork, synchronous, unchanged) and answers via SendMessage; the Manager resumes in place and continues. This replaces today's die-with-`ESCALATE`-verdict + `--resume` re-dispatch ceremony. The terminal `ESCALATE` verdict remains for genuinely terminal states (budget exhaustion, operator-gated stops); *interpretation questions become pauses, not deaths*.
- **Downward: the nudge.** Executor → Manager via SendMessage mid-campaign (course correction with context intact); Manager → specialists already rides the same mechanism. The operator watches any level live in the panel and steers by addressing the Executor — judgment escalates upward, direction flows downward, unchanged.

**4. Durability amendment — and the archive's rising role.** The Manager leaves the durable tier; its stamps anchor to the hosting Executor session (`manager@<machine>/<executor-session-id>`) — the existing "nearest resumable session" principle applied one tier up, no new grammar. Mitigation: **`session-archive` extends to sweep the parent's `subagents/agent-*.jsonl` files** alongside its transcript (MANIFEST entries included), so Manager forensics survive archival and a past Manager stays reachable by restoring + resuming its Executor. This lands just as the archive's purpose expands: per the operator's observation, Orchestra positions `work/sessions/` as **asynchronous observability for peer machines** — a peer reads a machine's archived hierarchy without any live process on it. Sweeping the subagent files makes the archived unit the *whole tree*, not just its root: exactly what a peer needs.

**5. Permission story.** With parent-mode dominance, the campaign's autonomy scope (decided at the Reconnaissance gate) maps to the **Executor session's own mode** rather than per-child `--permission-mode` flags. Where genuinely independent scoping is needed — a stricter-than-parent child, or beyond-depth dispatch — the headless child-session pattern remains documented as the **deliberate fallback**, demoted from default, its cause stated at the dispatch site.

## The spike (gates conversion — nothing in canon changes until it passes)

One work order; empirical probes against the installed CLI, results recorded in the order:

1. **Depth**: with the env var set, a dispatched subagent successfully dispatches its own subagent; record CLI version + exact behavior.
2. **Mid-turn nudge**: SendMessage to an actively-running subagent — delivery timing and interjection semantics (paused-agent delivery is documented; mid-run is the open question).
3. **Panel depth**: nested (depth-2) transcripts visible live to the operator.
4. **Durability**: locate `subagents/agent-*.jsonl`; end the parent session; `claude --resume <parent>` then continue the subagent by id — context intact.
5. **Model + permission**: per-call model honored at both depths; permission inheritance matches documented dominance.

Pass → `/plan` the conversion order. Any failure → reconciliation note here; the conservative fallback (child sessions + the gated accomplish-tail live-feed recipe) stands ready.

## Reconciliation note (26-07-24, mission v1 M3 — premise 1 inverted; the rest confirmed, and the boundary proved in anger)

Premise 1 is wrong, in the opposite direction from the one that would have hurt. It asserted that subagent nesting ships **off** on CLI v2.1.217+ and must be switched on with `CLAUDE_CODE_MAX_SUBAGENT_SPAWN_DEPTH`; the M1 probe battery ([`todos/_done/26-07-23.probe-battery.md`](../../todos/_done/26-07-23.probe-battery.md)), run against pinned **v2.1.218**, found nesting **default-on and deep** — root → depth-1 → depth-2 → depth-3, with no env var and no settings key anywhere — and found the variable to be a **restrictive cap**: `var=1` structurally withholds the Agent tool from a depth-1 child (no error, the capability is simply absent), `var=2` permits exactly the ladder this design needs. There was nothing to enable. The design item nonetheless survives the inversion whole, re-rationalized: the `.claude/settings.json` key `env: { CLAUDE_CODE_MAX_SUBAGENT_SPAWN_DEPTH: "2" }`, its BASELINE anchor and its `/initialize` audit all land as § The design specified them — no longer the gate that unlocks nesting, but a **pin and a cap**. It pins the depth the role ladder actually needs and caps blast radius against a platform that would otherwise nest without limit, which is the same blast-radius principle the design already invoked for the tight `2`, now doing real work instead of gatekeeping a door that stood open. The `/initialize` behavioral audit inverts with it, from *"is nesting on?"* to *"is the cap set as intended?"*. This is mission determination D2's re-scope, operator-confirmed 26-07-24 and landed in M2. The proposal's own standing lesson — that this capability must be **configured and audited, never assumed** — is vindicated again, this time by the premise being wrong in the other direction.

Premises 2 through 5 came through M1 confirmed as written: the observability inversion, native continuation across nesting levels, parent-scoped durability of the `subagents/agent-*.jsonl` transcripts, and parent-dominant permissions. One probe returned more than the premise claimed — nested **depth-2** transcripts stream live to the operator's panel, not merely the first level — so the observability case this whole proposal rests on holds at the depth of the tree, not just at its top edge.

The boundary was then validated in anger, before this note was written: M2's conversion campaign was itself run **through** the boundary it was converting. The load-bearing event ([`todos/_done/26-07-24.execute-conversion.md`](../../todos/_done/26-07-24.execute-conversion.md), phase 2) is that the phase-2 Manager subagent returned **without a verdict** — it had parked on a background child that had already completed. That is cause 4 of the premature-exit bug, and its order — [`todos/_done/26-07-23.bug.headless-manager-premature-exit.md`](../../todos/_done/26-07-23.bug.headless-manager-premature-exit.md) — had predicted precisely this outcome at filing: *"cause 4 survives mission v1's conversion — a subagent Manager can equally stop short of its verdict."* It did. What answered it came from here: the downward **nudge** of § The design item 3, a SendMessage to the parked Manager, which auto-resumed from its transcript with context intact and drove to a clean verdict with no re-dispatch and no lost work. The prediction was the bug order's; the remedy was this proposal's, and the mechanism validated itself during its own construction.
