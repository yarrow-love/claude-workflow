## Objective

- **Executor-held item (deliberate — the criterion-3 upward leg)**: the fold's **presentation on `main`** — true merge (`git merge 26-07-23` + purification commit; honest shared ancestry, the mission's blobs in `main`'s history) versus squash-fold (one synthetic commit; slim history, no shared ancestry with mission branches). **2. The fold recipe** — exact command series, written to be executed by the operator (or their session) after `/close`: fetch-fresh state assertions (`git status` clean, tag present), the merge (sole known conflict surface: `b332083` touched `.claude/` on `main` — pre-resolved instruction), the purification commit (enumerated `git rm` paths from the manifest — never `git add -A`), the template quality gate on the result (reference-resolution grep against the bare tree with the reconciled sanctioned-absence list; `bun` tooling smoke tests; `/initialize` audit), and the push. **3. The rehearsal must reproduce a `work/` tree that matches the reconciled BASELINE enumeration **exactly** (`diff` the file listing against the manifest's keep column). The runbook is instance content — it archives with the mission it closes.

## Reconnaissance amendments (26-07-24)

Appended at the campaign's Reconnaissance gate; milestone-architect confirmed. These **govern** where they conflict with the text above.

1. **`b332083` is not a conflict — the premise is falsified.** `git merge-tree --write-tree 26-07-23 main` exits 0 with three clean auto-merges (`BASELINE.md`, `commands/archive.md`, `commands/initialize.md`); the commit's touched lines are disjoint from the branch's 49 commits, and `HEAD..main` contains only that one commit, so no other surface exists. The "pre-resolved instruction" was protecting against *improvising mid-fold*, not predicting a real conflict. Replace it with a **pre-flight assertion**: re-run `git merge-tree --write-tree` immediately before the fold — phase 2 edits `BASELINE.md` again, so recon's verification will be stale — and **any conflict stops the ceremony for reassessment, never an ad-hoc resolution inside it.**

2. **Empty containers need `.gitkeep` *added*, not just contents removed.** `work/proposals/_enacted/` and `work/todos/_done/` are non-empty today so they carry no keeper; after the strip they would vanish from git entirely. Generalize rather than hand-enumerating: **any keep-column directory left with zero tracked files after the strip gains a `.gitkeep` in the purification commit.** Check `_deferred/` too — its content strips as well.
