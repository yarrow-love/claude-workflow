# /design — the design workflow, Designer role, and design-system management

> **Backordered 26-07-11**; **woken 26-07-12** when a live campaign (astrology M3 living-wheel) hit exactly the gap this order names — its Reconnaissance flagged "interactive design dialog vs autonomous comps" and "design-artifact home undecided" as open forks. Executed the same day; the operator supplied the promised exemplar.

## Feature Request

**What.** A `/design` command and a reconsidered Designer specification: how UI/UX design work enters the system, where design-system data lives, and how the Designer → Implementer → Inspector loop closes. Likely spawns a proposal before a plan.

**Why.** The operator is not satisfied with the current UI design workflow. The Designer agent today produces specs into a gap — there is no settled home for design-system artifacts in this repo's conventions, no `/design` entry point (design work rides `/plan`/`/execute` awkwardly), and the relationship between the Designer's verification criteria and the Inspector's pass/fail loop deserves deliberate design rather than the current convention-by-adjacency.

**Inputs to gather before planning:**
- The **`work/design/` convention** from the operator's prior projects — design-system data, design briefs, and pulls from Claude Design are stashed there. Supplied 26-07-12: `~/dev/lasers/src/design` (see Research Findings).
- Claude Design integration — how its outputs should land in `work/design/` and feed Designer/Implementer dispatches.
- The Designer and Inspector agent specs as they stand post-unification (renamed roster, browser MCPs now configured at user scope per `agentic-workflow/3.keystones.md`).
- Whether design work is a work-order `type` (rides `/execute`), its own command (`/design`), or both (a `/design` that files/advances a design-typed work order).

## Acceptance criteria

- A settled home and lifecycle for design-system data (`work/design/` or successor), documented in `work/PROJECT.md` + root conventions.
- The Designer → Implementer → Inspector loop specified end-to-end: who writes verification criteria, who consumes them, how FAIL feeds back.
- `/design` (or the decided entry point) lands in the command family without breaking the nine-command discipline — if it becomes command #10, the determinations record gets a dated amendment.

## Research Findings

> architect@play/a9ddd6b3-7457-451c-85bd-a79a91b05e08

**The exemplar** (`~/dev/lasers/src/design`, the operator's impromptu protocol): numbered hand-off briefs written for Claude Design (`briefs/NN-<slug>-brief.md`) with a proven internal shape — decided-design **"locks,"** a mental-model section, ASCII layouts, the token rule (*evolve values, keep names*), and a crisp **ownership boundary** ("mock layout and look, never behavior — Claude Design owns the visual system and arrangement; Claude Code owns behavior"); a `bundle/` hand-off package (tokens.css + research-distilled `principles.md` + `current-state/` png+html capture pairs per component); returned `.dc.html` comps loose at the container root; the push/pull lifecycle encoded only in commit messages.

**Assessment**: the *artifacts* are excellent (the brief shape and the bundle concept are adopted nearly verbatim); the *architecture* was improvised — authority via "master brief, superseded where conflicting" instead of a maintained canon; flat global numbering; lifecycle in prose and commit messages instead of document state; returns unlinked from their briefs; captures undated with invisible staleness; no verification criteria (that convention post-dates it).

**The architectural key** (settled in dialog): **design is judgment-closed** — the system's two closing modes already provide the shape. `/design` is an Architect-style dialog (operator taste is the convergence target); design *implementation* stays criteria-closed in normal work orders; taste review rides the existing operator-gated acceptance mechanism. No new mechanisms anywhere.

**Container shape** (three dialog rounds, operator-settled): `bundle/` → **`system/`** (the distilled living canon: tokens, glyphs, palettes), which split canon from evidence — as-is captures move into each hand-off's outbound snapshot. `briefs/` + `pulls/` → **`sessions/<YY-MM-DD>.<slug>/`** (one directory per round-trip exchange, named like work orders), flattened: **`BRIEF.md`** at the root as the exchange's running record (rejected: README.md — the caps name marks the defining artifact and earns the `**/BRIEF.md` glob), `current-state/` as the only outbound subdirectory, returns (index.html, assets, `components/<c>.html+.png`) directly at the hand-off root. Directional rule: BRIEF.md + current-state/ go out; everything else came in.

## Addendum (26-07-12) — organizational revision, same dialog

> architect@play/a9ddd6b3-7457-451c-85bd-a79a91b05e08

The operator reopened the organization after the first landing, and the revision is architecturally significant, not cosmetic: **the design container moves from `src/design/` to `work/design/`** — design is *process, and mission-versioned*, not product. It archives whole with the mission (`work/missions/<v>/design/`) and is triaged at the next settlement (**carried / redesigned / watch**, mirroring footguns), which honestly models the real lifecycle: MVP missions ignore UI design entirely (sanctioned — the container is a lazy-create of the founding `/design`), and later missions may revise it totally. The move also *fixed the location* — `work/design/` is now work-system convention, not a `work/PROJECT.md` variable (the supplement keeps only the external tool + the app-side token home). Corollary made explicit: **`system/` is authority, not build input** — the token file the app imports lives in app source and *conforms to* the canon, so archiving a mission can never break an app import. And **`hand-offs/` → `sessions/`**: with the reframe, the dated dirs record *design sessions* (dialog → brief → optional external hand-off phase → reconciliation), not just external exchanges — a design session with no external tool is equally valid. The "session" collision with the transcript archive is governed by a vocabulary rule in designer.md: unqualified "session" = Claude session; these are always "design sessions." Mission-close (`/close`) and settlement (`plan-mission`) gained the design-triage steps.
