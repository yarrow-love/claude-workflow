### Spec-readiness (researcher) — gaps to close before launch

The probe *procedures* are concrete; the *dispatch content and pass thresholds* are not. To make every probe reproducible-verbatim by an operator, the seeded work order must pin:

1. **Literal dispatch prompts** for probes 1, 2, 5 — each ending "**read-only; do not write or edit any file**" (there is no permission backstop under `bypassPermissions`; prompt discipline is the only guardrail).
2. **A fixed harmless read target** — one small named file every probe subagent reads and reports (reproducible, non-mutating, acceptance-irrelevant).
3. **Probe 2 long-running mechanism** — a subagent task that runs long enough to interrupt (sequential multi-file read), so the mid-turn nudge lands before natural completion.
4. **Probe 4 planted fact** — a specific fact worded into an *earlier* probe's dispatch prompt (probe 1 or 5), which the resumed subagent is later asked to recall. Without a planted fact, "context intact" has no concrete check. Wire this dependency explicitly.
5. **Two named models for probe 5** — depth-1 vs. depth-2, distinct and self-identifiable.
6. **Verdict-recording cadence** — write each verdict to the work order *immediately on completion*, not batched (crash-safety, currently stated only inside probe 4).
7. **Failure-path recipe** — on probe 1/4 fail: stop, no workaround, write the fail verdict + observed behavior, hand back to this Executor session for escalation per MISSION failure criteria.
8. **Premise-reconciliation table skeleton** — 5 pre-drafted rows (premise → confirmed/diverged) so close is fill-in, not improvisation.

### Probe-4 / mechanics risk — assessed, low, one mitigation

- **No consult collision.** Probe 4's `claude --resume <parent>` *without* `--fork-session` is a distinct code path from `consult` (which *always* forks); it is the *same* in-place-continuation pattern the executor doc already documents for Manager re-dispatch. Not novel, not risky.
- **Archive-on-stamp gap (the one real mitigation).** `session-archive` (SessionEnd hook) only archives a session whose id is stamped somewhere under `work/`. Ending the probe-parent mid-battery for probe 4 earns *no durable archive* unless its id was stamped first — so if the bare resume misbehaves, there's nothing in `work/sessions/` to restore from. **Mitigation: the operator stamps the probe-parent session id into this work order's `sessions:` frontmatter before ending it** (also correct per callback grammar — the probe-parent is a durable Manager-tier session). The `subagents/agent-*.jsonl` sweep is **M2/D3 scope, not built yet** — probe 4 records live-cache paths only; note that they could rotate before M2 lands the sweep (low risk for a same-day spike).
- **`commit-guard` (Stop hook)** fires when the probe-parent stops; harmless here (probes don't close `_done` docs mid-run) but may nag once if uncommitted verdict text matches its patterns — expected, not a fault.
