# Commit follow-through — approved work must land without operator nudging

## Feature Request

**What / why**: The operator repeatedly has to nudge sessions to commit finished, approved artifacts. Observed 26-07-23: the M1 milestone-architect session hardened the milestone correctly, paused for sign-off per protocol — and never performed its step-7 close-out (commit + transcript archive) after approval; the mission architect committed on operator instruction. The operator's requirement: *"the operator should not have to habitually nudge the agent to follow through by committing work."*

**Diagnosis — the mandate exists; the follow-through breaks at the approval turn-boundary.** `plan-milestone.md` step 7, `plan-mission.md` step 6, and the manager's Closing Out all mandate commit-and-archive. The failure mode is structural: "present → await approval → commit" splits across turns, and the post-approval turn treats the approval as the ending instead of the trigger. A doc that says "commit" at the end of a numbered list does not survive that boundary.

**Two remedies, both in scope:**

1. **Doctrine — the follow-through rule, stated once and bound per-role.** On receiving sign-off, the session's *first actions* are the mechanical close-out — commit (enumerated paths), transcript archive, status/roadmap updates — *before* any reply prose; a turn never ends with approved-but-uncommitted artifacts; while awaiting sign-off, the presentation explicitly states "uncommitted pending your sign-off" so the state is visible rather than ambient. Home: one canonical statement (`work/README.md` or the agents' shared conventions) + a constraint line in `architect/CLAUDE.md` and `manager/CLAUDE.md`; sweep every workflow's terminal step to reference it rather than restating.
2. **Harness net — a `Stop` hook as the backstop.** A small `.claude/bin/` script wired in `settings.json`: on session stop, if `git status --porcelain` shows dirty/untracked paths under `work/` or `.claude/`, feed the agent a blocking reminder — "uncommitted workflow artifacts: <paths> — commit them, or state to the operator why they remain uncommitted (awaiting sign-off is a valid, stated reason)." Deliberate behavior, not silence; the valid-pause case stays legitimate but becomes *explicit*. Project-generic; BASELINE-manifested; `/initialize`'s hooks-resolve check covers it.

**Constraints**: genericity (nothing project-specific in the hook); the hook must not block legitimately-paused sessions (awaiting sign-off, mid-dialog) — remind-and-require-statement, not hard-fail; commit conventions unchanged (enumerate paths, prefixes, Co-Authored-By).

**Acceptance**:
- Terminal-step sweep: every workflow that authors artifacts ends by invoking the follow-through rule; the rule exists in exactly one canonical home.
- The Stop hook exists, resolves (`/initialize` check 3), and demonstrably fires: a session stopped with a dirty `work/` file receives the reminder; a clean stop stays silent.
- BASELINE manifest + settings skeleton updated in the same change.
