## How to run this order

**Entry state: no planning pass.** This order arrives with a complete, operator-blessed procedure (the probe battery below) — it enters directly at execution. Do not decompose to the specialist roster.

**The executing session IS the probe parent.** The operator launches an interactive session with the depth env var set and runs this order inside it; that session dispatches the probe subagents *itself* (its own depth-1 Agent-tool calls, which in turn dispatch depth-2). Delegating the probes to an implementer subagent would insert a layer and change the depth topology under test — so the main session runs them directly.

**Operator-gated (Step 0).** Deferred until the operator has a clean window: the CLI upgrade and the whole run are the operator's steps. Nothing runs headless.

### Standing rules for the whole battery

- **Read-only probe subagents.** Every dispatch prompt below ends "read-only; do not write or edit any file." There is no permission backstop under this repo's `bypassPermissions` — prompt discipline is the only guardrail. Run `git status --porcelain` before probe 1, before probe 4, and at close; the tree must show only this work order + milestone docs dirty.
- **Fixed harmless read targets.** Probe subagents read only these existing tracked files and report a token from them: `work/milestones/1.spike/MILESTONE.md` (report the `# M1 — …` title line) and `work/PROJECT.md` (report its first heading). No other file access needed.
- **Write each verdict to this order immediately on completion** — not batched. Crash-safety applies after *every* probe, not just probe 4.
- **Failure path (probes 1 & 4 are load-bearing).** On a fail: stop the battery, do **not** attempt a workaround, write the fail verdict + exact observed behavior to this order, and hand back to the Executor session (`executor@play/a52d1822-…`) for escalation per [MISSION](../MISSION.md) failure criteria. A probe-1/4 fail closes the milestone by escalation, never by working around it.
- **The planted codeword** for probe 4's context-retention check is **`ORCHID-7`** — planted in the depth-1 probe-5 subagent (below), recalled after the end/resume. A synthetic codeword (not a file fact) is used deliberately: it cannot be re-derived by re-reading a file, so recalling it proves *in-run context* survived, not just re-lookup.

## Probe battery

Execution order (fail-fast on load-bearing, panel-attestation continuous, session-ending last): **1 → 5 → 2 → 3 → 4**.

### Probe 1 — Depth (gates the battery)

**Control arm** — in a *separate* session launched **without** the env var (`claude`, no prefix):
- Dispatch prompt (depth-1): *"You are a depth-1 probe. Attempt to dispatch a subagent (via the Agent/Task tool) that reads `work/PROJECT.md` and reports its first heading. Report back: did the Agent tool permit the nested dispatch, or were you refused/unable? Quote any error verbatim. Read-only; do not write or edit any file."*
- **Expected:** nested dispatch unavailable/refused (nesting off by default ≥v2.1.217). Record the exact behavior + `claude --version`. This control is the behavioral check M2's `/initialize` audit inherits.

**Gated arm** — in the env-var probe parent:
- Dispatch prompt (depth-1): *"You are a depth-1 probe. Dispatch a depth-2 subagent (via the Agent/Task tool) with exactly this instruction: 'Read `work/milestones/1.spike/MILESTONE.md` and report the exact milestone title (the line starting `# M1 —`) back to your launcher. Read-only; do not write or edit any file.' When the depth-2 subagent reports, relay its exact answer to me and state your own agent identity. Read-only; do not write or edit any file."*
- **Pass:** the depth-2 subagent completes and returns the correct title; record the chain (depth-1 id → depth-2 id → returned title) + `claude --version`.
- **Fail here → stop the battery, escalate.**

### Probe 5 — Model + permission (2nd; plants the codeword)

**Model arm** — per-call model override at both depths (D-d: opus at depth-1, sonnet at depth-2):
- Dispatch prompt (depth-1, per-call `model: opus`): *"You are a depth-1 probe running under a per-call model override, and you must retain this context — you may be resumed later. Do all of the following: (1) State which model you are running as, from your own system prompt/self-knowledge. (2) Commit this codeword to memory: **ORCHID-7** — you will be asked to recall it after a session restart. (3) Dispatch a depth-2 subagent with a per-call model override of `sonnet`, instructing it: 'State which model you are running as, then read `work/PROJECT.md` and report its first heading back to your launcher. Read-only; do not write or edit any file.' (4) Relay the depth-2 subagent's exact model self-report and its answer back to me. Read-only; do not write or edit any file."*
- **Record:** depth-1 self-reports **opus**, depth-2 self-reports **sonnet** (quote both self-reports as evidence); note the **depth-1 subagent's agent id** — probe 4 resumes *this* subagent to check codeword recall.
- Adapted-with-note if a self-report is ambiguous or the depth-2 override is not honored (names proposal premise 5).

**Permission arm** — parent-mode dominance (confirmed by docs; observe the structural fact):
- Under the `bypassPermissions` probe parent, a child *cannot* be made stricter — docs: a parent's `bypassPermissions`/`acceptEdits` "takes precedence and can't be overridden," child frontmatter ignored. Record what was attempted (if the harness exposes any lever to request a stricter child `permissionMode`) and what was observed. If no lever exists to even attempt a stricter child, **record that as the observation** — dominance is structural: there is no way to make a child stricter. Verdict: pass if observed behavior matches documented dominance; adapted-with-note (naming premise 5) if the only observable is "no lever exists."

### Probe 2 — Mid-turn nudge (3rd)

**Running arm** — mid-turn delivery to an actively-running subagent:
- Dispatch prompt (depth-1, long-running): *"You are a depth-1 probe. Read these files one at a time, and after each output a one-line summary before moving to the next: `work/README.md`, `work/PROJECT.md`, `work/MISSION.md`, `.claude/BASELINE.md`, `work/milestones/README.md`, `work/todos/README.md`. Report each summary as you finish it, one at a time. Read-only; do not write or edit any file."* (The sequential reads give a window to interject before natural completion.)
- **While it is still reading**, SendMessage it: *"Also report the total number of files you were asked to read, and stop after the current file."*
- **Record:** delivery timing (roughly how quickly it lands) and interjection semantics — documented expectation (v2.1.198): the launcher's mid-run message is treated as normal task direction. Note whether it acted mid-run or only at the next turn.

**Completed arm** — for contrast:
- After a depth-1 subagent has *finished*, SendMessage it a follow-up (*"Now also report the first heading of the last file you read."*). **Record:** whether it auto-resumes (documented expectation: a completed subagent auto-resumes in the background without a new Agent invocation).

### Probe 3 — Panel depth (4th; operator attestation, not a dispatch)

The operator attests, **in this order**, what was *live-visible in the panel* during probes 1 / 5 / 2:
- Were depth-1 **and depth-2** transcripts visible live (streaming, or only a tree/`(+N)` count)?
- Was task/status visible at each depth?
- Did any permission prompt from depth-2 surface to the main session?

This is the load-bearing *undocumented* observation (docs cover the tree view and depth-1 permission surfacing; depth-2 live streaming is unspecified). Write the attestation as prose evidence. Verdict: pass / adapted-with-note (naming premise 2) per what was actually visible.

### Probe 4 — Durability (last; ends the parent)

1. **Record the subagent transcript paths** for the probe tree — shape `~/.claude/projects/-home-yarrow-dev-claude-workflow/<parentSessionId>/subagents/agent-<agentId>.jsonl` — especially for the **depth-1 probe-5 subagent holding `ORCHID-7`**. (Note: the `session-archive` sweep of these files is M2/D3 scope, not built yet — these are live-cache paths and could rotate before M2 lands the sweep.)
2. **Write all verdicts gathered so far** to this order (crash-safe).
3. **Stamp the probe-parent session id** into this order's `sessions:` frontmatter as `manager@play/<probe-parent-sessionId> # probe parent` (D-c) — *before ending*. This earns the archive-on-stamp copy in `work/sessions/`, the restore fallback if the bare resume misbehaves.
4. **End the parent session.**
5. **Resume it in place:** `claude --resume <probe-parent-sessionId>` — deliberately **without** `--fork-session` (continuation *is* the probe; same in-place pattern the executor doc uses for Manager re-dispatch).
6. **Continue the depth-1 probe-5 subagent by id** via SendMessage: *"Recall the codeword I asked you to remember earlier, and state it."* **Pass:** it answers **`ORCHID-7`** — context intact across the end/resume (proves in-run memory, not file re-lookup). Record the probe-4 verdict from the resumed session.
7. **Contingency:** if `claude --resume` errors outright, fall back to a `work/sessions/` restore (enabled by the step-3 stamp) or record probe-4 **fail** + escalate.
