# /configure — guided capability installation (the repair half of /initialize)

> **Backordered 26-07-11** by the operator — deliberately parked until a ripe opportunity to focus on the configure/initialize pair. Consult handle for the design context: the `architect@play/…` entry in `sessions:`.

## Feature Request

**What.** A `/configure <capability>` command that augments the baseline Claude configuration with project- or environment-specific capabilities — installing MCP servers (`/configure Sanity MCP`, `/configure playwright`), scoping them correctly (user vs project `.mcp.json`), adding the permission allowlist entries they need, and updating `work/PROJECT.md` / `.claude/BASELINE.md` expectations to match.

**Why.** MCP/capability setup is a confusing, distracting process for the operator. `/initialize` (landed in `agentic-workflow/5.provenance.md`) *detects* — every failure in its audit report names a fix; `/configure` is the counterpart that *performs* the fix. The pair completes the bootstrap story: copy baseline → `/initialize` → `/configure` the gaps.

**Design notes from the determinations dialog (26-07-11):**
- Consumes `/initialize`'s audit output — "missing: Playwright MCP → run `/configure playwright`".
- Mutating command touching user-level config (`claude mcp add --scope user`), project `.mcp.json`, and `settings.json` permission allowlists — deserves its own design pass rather than landing as a side-effect of the unification series; that is why it was deferred.
- Should carry guided flows for the recurring servers (Context7, Playwright, Chrome DevTools, Sanity) with their exact add-commands sourced from `.claude/BASELINE.md` recipes.
- Consider: expanding `/initialize` in the same pass (deeper audits: worktree hygiene, model-pin availability) — the two commands are a detect/repair pair and should be designed together.

## Acceptance criteria (provisional — harden at /plan)

- `/configure <known-capability>` installs, verifies (`claude mcp list` or equivalent), updates the relevant manifest/supplement, and reports.
- `/configure` with no argument lists known capabilities and their install state (delegating detection to the `/initialize` audit).
- Every mutation it performs is enumerated in its report; nothing silent.
