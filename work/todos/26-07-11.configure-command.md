# /configure ‚Äî guided capability bootstrap (the repair half of /initialize)

> **Backordered 26-07-11**; **woken and re-scoped 26-07-12** by the operator when the `/design` hand-off protocol made the capability gap concrete: a project agent must never burn context discovering or working around missing transport. Consult handle for the design context: the `architect@play/‚Ä¶` entry in `sessions:`.

## Feature Request

**What.** A `/configure <capability>` command that augments the baseline Claude configuration with project- or environment-specific capabilities, across **three capability classes**:

1. **MCP servers** ‚Äî install at the right scope (`claude mcp add --scope user` vs project `.mcp.json`): `/configure playwright`, `/configure Sanity`.
2. **Harness-tool authorizations** ‚Äî capabilities that need *authorization, not installation*: the **DesignSync** design scopes on the claude.ai login (verify with a `list_projects` read; headless sessions route through `/design-login` where the harness offers it). The design hand-off is the driving case: `/configure design` verifies DesignSync reachability, runs the **project-linkage flow** (`list_projects` ‚Üí pick with the operator, or `create_project`; verify `PROJECT_TYPE_DESIGN_SYSTEM`; record `design-project: <id>` in `DESIGN.md` frontmatter), and updates `work/PROJECT.md`.
3. **Permission allowlists** ‚Äî the `settings.json` entries a capability's commands need, so headless children don't auto-deny.

Every flow ends by updating the relevant expectations (`work/PROJECT.md`, `.claude/BASELINE.md`) so `/initialize` can audit what `/configure` configured.

**Why.** MCP/capability setup is a confusing, distracting process for the operator. `/initialize` (landed in `agentic-workflow/5.provenance.md`) *detects* ‚Äî every failure in its audit report names a fix; `/configure` is the counterpart that *performs* the fix. The pair completes the bootstrap story: copy baseline ‚Üí `/initialize` ‚Üí `/configure` the gaps.

**Design notes from the determinations dialog (26-07-11):**
- Consumes `/initialize`'s audit output ‚Äî "missing: Playwright MCP ‚Üí run `/configure playwright`".
- Mutating command touching user-level config (`claude mcp add --scope user`), project `.mcp.json`, and `settings.json` permission allowlists ‚Äî deserves its own design pass rather than landing as a side-effect of the unification series; that is why it was deferred.
- Should carry guided flows for the recurring servers (Context7, Playwright, Chrome DevTools, Sanity) with their exact add-commands sourced from `.claude/BASELINE.md` recipes.
- Consider: expanding `/initialize` in the same pass (deeper audits: worktree hygiene, model-pin availability) ‚Äî the two commands are a detect/repair pair and should be designed together.

## Acceptance criteria (provisional ‚Äî harden at /plan)

- `/configure <known-capability>` installs or authorizes, verifies (`claude mcp list` / a DesignSync read / equivalent), updates the relevant manifest/supplement, and reports.
- `/configure design` completes the full hand-off bootstrap on a fresh project: DesignSync verified, project linked (or created), `design-project:` recorded, PROJECT.md updated ‚Äî leaving `/design`'s hand-off phase zero-discovery for the project agent.
- `/configure` with no argument lists known capabilities and their install/authorization state (delegating detection to the `/initialize` audit).
- Every mutation it performs is enumerated in its report; nothing silent.
