# `.claude/bin/consult` forks are autonomous agents that mutate the shared working tree — cross-project, concurrently

## Bug Report

**Symptom (witnessed live 26-07-23, during the `commit-guard-context-tier-quieting` order).** A background consult **from a different project** (`~/dev/orchestra`) forked the *shared* architect session `76ba13c6@play` and executed autonomously **inside `~/dev/claude-workflow`'s working tree**, making git commits and branch checkouts in a repo it does not own. Reconstructed evidence (reflog will age out — captured here):

- Rogue process (from `ps`, PID 1607428, since exited):
  ```
  cd /home/yarrow/dev/claude-workflow && bun .claude/bin/consult \
    "architect@play/76ba13c6-e6f5-4933-9e6c-7ef3fbdd6b7d" \
    "$(cat /tmp/claude-1000/-home-yarrow-dev-orchestra/…/scratchpad/consult-genericity.txt)" \
    --model fable > /home/yarrow/dev/orchestra/tmp/accomplish/consult-genericity.out
  ```
  An **orchestra**-project consult, output redirected into **orchestra's** tmp, but `cd`'d into and operating on **claude-workflow's** git.
- Reflog trace of what that fork did: committed `e56dab3` ("Fix: genericity …") onto branch `26-07-23`; `checkout` main; `cherry-pick` the same change onto main as `b332083`; branch switched back to `26-07-23`. i.e. it committed to a feature branch **and to trunk**, unreviewed.
- Concurrently, a consult fork spawned by *this* session (also forking `76ba13c6`) committed `8ca7bdb` to break its own `commit-guard` re-fire loop.

**Two distinct defects compound here:**

1. **Consult forks are not advisory — they are full autonomous agents.** `.claude/bin/consult` runs `claude -p --resume <id> --fork-session`. The Planner/Architect callback policy frames consults as "advisory, read-only; distill into the work doc." But the fork inherits full tool access and the invoking shell's cwd, so it can (and did) `git commit`/`checkout`/`cherry-pick`. Nothing sandboxes it to read-only.
2. **A shared session handle is forkable concurrently from multiple projects, into the wrong working tree.** `76ba13c6@play` is machine-local and both claude-workflow and orchestra resumed/forked it at once. The orchestra fork `cd`'d into claude-workflow. This directly violates the intended **one-way valve** of the orchestra spin-out (orchestra must not reach back into the parent) — see the "Orchestra handoff" memory.

**Impact.** Unreviewed commits landed on a feature branch (`e56dab3`) and on **trunk** (`b332083`); a session's working tree mutated between its own turns (phantom dirty `.claude/BASELINE.md`, `archive.md`, `initialize.md` appearing then vanishing); two sessions (this manager + an architect consult) were partly trapped in the `commit-guard` loop because the tree kept changing under them; risk of clobbering uncommitted work via the concurrent `git checkout main`/back.

**Where.** `.claude/bin/consult` (the `--fork-session` invocation and its cwd/tool inheritance); the cross-project handle-sharing is a machine-level concern (the `play` machine hosts both repos and both can resume `76ba13c6`). Related standing context: the orchestra one-way-valve intent; `.claude/agents/manager/CLAUDE.md` § Planner Callback (which asserts consults are advisory — the code doesn't enforce it).

**Note on `e56dab3`/`b332083`.** The genericity change itself is correct and on-mission (drops instance-history `work/` paths from the portable `.claude/` set) and was **kept** (operator-agreed 26-07-23). This bug is about the *mechanism*, not that commit's content. `b332083` sits directly on `main` — flagged for the operator's awareness.

**Needs** (for a future `/execute` — no resolution attempted here): likely an architect-tier design pass. Candidate directions to weigh, not decide: consult forks run read-only (strip write tools / run in a throwaway worktree or checkout); consult refuses when the resolved session's repo ≠ the invoking cwd's repo; the cross-project valve is enforced (a handle records its owning repo; foreign repos degrade to documents, never fork-in-place).

## Investigation Findings

Root cause traced across every consult call site; harness capabilities verified against current docs (claude-code-guide). **The defect is the total absence of a capability boundary on an autonomous fork — enforceable today with existing flags plus one cheap local check.**

**RC1 — `consult` spawns a full-capability, cwd-inheriting agent.** `.claude/bin/consult:79,86` builds `["claude","-p","--resume",uuid,"--fork-session","--model",model,prompt]` and `spawnSync(…, {stdio:"inherit"})` with **no `cwd` option** (→ inherits the caller's cwd) and **no** `--permission-mode`/`--allowedTools`/`--disallowedTools`/`--strict-mcp-config`. The "advisory, read-only" claim (`consult:12`, Planner Callback policy) is prose with zero code enforcement. **Blast radius is every consult site**, not just the script: four raw `claude -p --resume --fork-session` invocations bypass it entirely — Planner Callback (`manager/CLAUDE.md:123`), escalate-on-novelty (`execute-work-order-automatically.md:14`), ESCALATE architect (`execute-milestone.md:19`), and the **mandatory Architect Blessing gate** (runs on *every* work order).

**RC2 — no repo-binding check; the needed fact is already on disk.** `consult:21-26` derives `repoRoot` from the invoking cwd (`git rev-parse`) and finds the transcript under that cwd's project-slug — it never asks whether it *should* fork this handle from here. The orchestra incident succeeded because the rogue command `cd`'d into claude-workflow first. **A "refuse on repo mismatch" check is cheap and local**: every transcript line carries a `"cwd"` field recording the session's true origin (verified: `76ba13c6…jsonl` → `"cwd":"/home/yarrow/dev/claude-workflow"`); compare `resolve(that)` to `resolve(repoRoot)`, refuse/degrade on mismatch. No new registry needed. (Deeper: the callback grammar is role/machine/session-id only — **no repo dimension** — so a handle is machine-wide-forkable by construction; making it repo-qualified is a canon-level option.)

**RC3 — `bypassPermissions` inherited fully; but CLI flags override it.** `settings.json:2-5` sets `defaultMode: bypassPermissions`; the fork inherits it (every tool auto-approved, no prompt to intercept). **However** — deny wins at any level: CLI `--disallowedTools`/`--permission-mode` and inline `permissions.deny` beat the project default. The harness offers `--permission-mode plan` (blocks all writes in headless) and `--allowedTools "Read,Grep,Glob,WebFetch"` (whitelist → read-only). **A fully-worked three-layer hardening recipe already exists in this repo** — `work/todos/26-07-22.lan-intercom.md:96` (inline `permissions.deny` + `--disallowedTools` + `--permission-mode dontAsk`; `--strict-mcp-config` + empty `--mcp-config` for **MCP severance** — else an allowlisted mutating MCP tool runs unprompted; `WebFetch`/`WebSearch` denial for exfiltration) — but it was scoped to a *halted remote server*, never applied to the local, always-invoked script.

**RC4 — concurrency: no mutual exclusion on the shared tree.** `--fork-session` forks the transcript, not the working tree; two forks inherit the same cwd and race one `.git` (the reflog's checkout-main→cherry-pick→checkout-back interleaved with the other fork's commit). No lock, no worktree isolation (the Manager's `isolation:"worktree"` exists but isn't wired to consult), no clean-tree precondition.

**Exists vs. must-build** (for the planner): *exists* — the `cwd` provenance data (unused), the three-layer hardening recipe (on paper, remote-scoped), `consult`'s own `repoRoot` computation, the `isolation:"worktree"` precedent. *Must build* — the repo-mismatch comparison in `consult`; applying/adapting the hardening to the **local** path; any concurrency control; optionally a repo dimension in the callback grammar. **Load-bearing constraint**: the Blessing gate is a consult-fork on every work order's critical path — read-only enforcement must preserve *read + return a verdict* while removing *act*. **Related halted order**: `work/todos/26-07-22.lan-intercom.md` carries much of this design (and a `--fork-session` sweep a repo-binding fix would also touch) — the planner should weigh extracting a **shared hardening primitive** both paths use vs. duplicating.

## Resolution Plan

Two honesty notes reshape the design up front: **(a) `deny` is load-bearing, not belt** — this repo's `settings.json` allows `Bash(claude:*)`, and only a `deny` rule defeats a project `allow`, so a mode/allowlist-only fork could still spawn an unconstrained child `claude`. **(b) The repo-binding guard would not have caught the witnessed incident** (the rogue command `cd`'d in first → cwd matched); it closes the *transcript-provenance* hole (a foreign transcript forked into a local tree), while **Phase 1's read-only boundary neutralizes the foreign-invoker case** — the worst a `cd`'d-in consult can now do is read and answer. True valve enforcement is machine-level (Out of Scope).

**Phase 1 — enforced read-only fork boundary in `.claude/bin/consult`** (the `cmd` array ~L79, `spawnSync` ~L86, header). **Five** independently-verified layers, each covering a hole another leaves: inline `--settings {permissions:{defaultMode:"dontAsk",deny:DENY}, disableAllHooks:true}` (the wall — `deny` binds even under `bypassPermissions` and beats `allow`); `--disallowedTools DENY` (same list, CLI layer — redundant by design, two mechanisms must both regress); `--permission-mode dontAsk` + `--allowedTools "Read,Grep,Glob"` (default-deny posture, covers tools no deny list can pre-enumerate); `--strict-mcp-config --mcp-config {mcpServers:{}}` (MCP severance — a deny list can't enumerate user-scope mutating MCP tools like Sanity/Higgsfield); **`disableAllHooks: true` (Blessing D1 — the fifth layer)**: hooks execute repo-configured shell *outside* all permission layers, so without this the fork inherits this repo's `Stop`→`commit-guard` (which, on a dirty close-out tree — the normal mid-work-order state when Blessing consults run — would block the fork and tell it to commit, the witnessed `8ca7bdb` shape with the escape welded shut) and `SessionEnd`→`session-archive` (a write into `work/sessions/` from a "read-only" fork, benign *today only by the accident* that archive-on-stamp gates on a stamped uuid). `DENY = [Edit,Write,NotebookEdit,Bash,Task,WebFetch,WebSearch]` (`Task` so the fork can't route around via a subagent; `WebFetch/WebSearch` as an exfiltration channel from a poisoned transcript). Pin `spawnSync(..., {cwd: repoRoot})`. Build-time-verify the `disableAllHooks` setting name against the installed binary alongside the other flag checks. **Blessing preserved**: `Read/Grep/Glob` stay — the architect fork reads the order and returns a verdict, losing only the ability to *act* (no git/write/shell/web/MCP/hooks; cost: no `git log` in-fork, but blessings are document-reads).

**Phase 2 — repo-binding guard, same file** (between the transcript-resolution ladder ~L59-73 and the prompt build). Read the target transcript's recorded `cwd` — the **first line carrying a `cwd` field** (Blessing D2 — *not* "first parseable line": transcripts often lead with a `summary` entry that has no `cwd`, which would silently degrade the guard to warn-and-proceed on exactly the transcripts it could bind); cached path reads `localPath`, archive path parses from the gunzipped buffer *before* materializing (so a foreign transcript never pollutes the local cache); refuse if it isn't `resolve(repoRoot)` or a subpath. **Refuse, exit 4** (1/2/3 taken), house-style message naming the origin repo + the degrade path ("consult from the owning repo, or answer from documents + a fresh dispatch"). **No `--force-foreign` override** (the hole with a nicer name). Missing `cwd` field → warn and proceed (Phase 1 bounds the damage).

**Phase 3 — route all consult sites through the script** (`manager/CLAUDE.md` Planner Callback L123 + Architect Blessing L68; `execute-work-order-automatically.md:14`; `execute-milestone.md:19`). All four carry the raw `claude -p --resume --fork-session` recipe with "prefer `.claude/bin/consult` once available" — it's available and now the *only* invocation with the boundary, so a raw recipe is a documented bypass. Replace with `bun .claude/bin/consult "<handle>" "<question>"`, keep each site's semantics, demote flags to a parenthetical. Grep `.claude/` for residual raw consult recipes (the Executor *continuation* re-dispatch at `execute-milestone.md:18` is NOT a consult — deliberately left). *Why route-through, not harden-each*: duplicated security invocations across prose are the exact drift engine that shipped the bare path while the hardening sat on paper.

**Phase 4 — canon note + lan-intercom coordination** (`work/README.md` § callback grammar; `26-07-22.lan-intercom.md` HOLD banner). Add two *property* sentences to the conventions bullets (consults are read-only by construction; consults are repo-bound) — **no repo segment added to the handle** (Decision 6). Annotate the halted lan-intercom order: its Phase-2 local boundary now lives in `.claude/bin/consult`; the server inherits it on unhalt, and its client exit taxonomy (4-7) renumbers around the now-taken exit 4.

**Phase 5 — adversarial verification (the boundary isn't trusted until witnessed).** Scratch clone with a *maximally adversarial* `.claude/settings.json` (`bypassPermissions`, `allow: ["Bash(*)"]`), transcript `cwd` rewritten to pass the guard for capability probes. Probes: **A** denied-write (tree clean, hash unchanged); **B** denied-commit/shell — the bug's exact shape, `git commit`/`checkout`/`touch` under `Bash(*)` allow → HEAD/reflog/branches unchanged (proves deny beats allow); **C** MCP severance (zero tools); **D** repo-binding (original cwd → refuse exit 4); **E** blessing viability (shape the prompt as a miniature blessing — Read a doc, return a structured verdict → success); **F** transcript immutability (byte-identical); **G (Blessing D1)** hook severance — the adversarial scratch clone carries a canary `Stop`/`SessionEnd` hook that writes a file; assert it never fires. Then the full quality gate. **Probes B, D, and G are the acceptance criteria for the root causes.** Build-time: verify the flag syntax against the *installed* binary first (separators, `dontAsk` validity, inline `--settings`/`--mcp-config` forms, the `disableAllHooks` setting name).

**Concurrency (RC4) — no phase, deliberately.** Enforced read-only *subsumes* the race — **conditional on Phase 1's hook severance (D1): hooks were the residual mutating path.** With them severed and Bash-git denied, the reflog interleaving the incident showed is impossible and two read-only forks can corrupt nothing. Residuals (stale mid-mutation read; identical double-write of a restored archive into `~/.claude/projects/`, outside the repo) are benign. No lock/worktree/clean-tree precondition — revisit only if a read-write fork class is ever introduced.

**Execution order**: single implementer, sequential — Phase 1+2 (one file, one commit) → Phases 3 & 4 (file-disjoint docs) → Phase 5 gates close-out. A failed probe reopens Phase 1/2, not the docs.

### Decision points

1. **Four-layer defense-in-depth** *(rec)* vs. a single minimal layer. `deny` is the only class that defeats this repo's `Bash(claude:*)` allow; `dontAsk` covers unenumerable future/MCP tools. Cost: longer invocation, fork loses read-only MCP.
2. **WebFetch/WebSearch denied** *(rec)* vs. allowed for consult-time research. Outbound fetch from a full-context fork is an exfiltration channel triggerable by a poisoned transcript; consults answer from held context + repo reads.
3. **Repo mismatch → refuse (exit 4), no override** *(rec)* vs. in-script auto-degrade or `--force-foreign`. The script can't perform the degrade (agent behavior); an override reintroduces the hole.
4. **Route all sites through the script** *(rec)* vs. harden each raw recipe. Makes the script mandatory infra on the Blessing critical path — its smoke test + Phase 5 probes are what make that safe.
5. **No separate concurrency control** *(rec)* — read-only forks can't race `.git`.
6. **Callback grammar: annotate the repo-binding property; do NOT add a repo segment to the handle** *(rec — canon-level, operator call)* vs. `role@machine:repo/uuid`. The transcript `cwd` is authoritative provenance already; a fourth segment is redundant and churns every existing stamp.
7. **Shared hardening primitive deferred** *(rec)* — boundary lives in `consult`; annotate lan-intercom to inherit on unhalt, vs. extract a `_lib.ts` now for a consumer that may never build.

**Out of scope** (seed follow-ups): the **foreign-invoker hole** (a process `cd`-ing in before consulting — capped at read-and-answer by Phase 1; true fix is machine-level caller identity, the intercom's territory); exit-taxonomy reconciliation with lan-intercom's client codes; the `repoRoot`/`slug` boilerplate `_lib.ts` de-dup (already seeded); `b332083` on `main` (operator-kept, noted). **Genericity**: pure CLI + inline JSON, no machine/project facts; canon note is property-only; no BASELINE change (no files added/moved/removed).
