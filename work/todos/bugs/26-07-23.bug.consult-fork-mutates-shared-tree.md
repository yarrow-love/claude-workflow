# `.claude/bin/consult` forks are autonomous agents that mutate the shared working tree — cross-project, concurrently

## Bug Report

**Symptom (witnessed live 26-07-23, during the `commit-guard-context-tier-quieting` order).** A background consult **from a different project** (`~/dev/orchestra`) forked the *shared* architect session `76ba13c6@play` and executed autonomously **inside `~/dev/claude-workflow`'s working tree**, making git commits and branch checkouts in a repo it does not own. Reconstructed evidence (reflog will age out — captured here):

- Rogue process (from `ps`, PID 1607428, since exited):
  ```
  cd /home/yarrow/dev/claude-workflow && bun .claude/bin/consult \
    "architect@play/76ba13c6-e6f5-4933-9e6c-7ef3fbdd6b7d" \
    "$(cat /tmp/claude-1000/-home-yarrow-dev-orchestra/…/scratchpad/consult-genericity.txt)" \
    --model fable > /home/yarrow/dev/orchestra/tmp/accomplish/consult-genericity.out
  ```
  An **orchestra**-project consult, output redirected into **orchestra's** tmp, but `cd`'d into and operating on **claude-workflow's** git.
- Reflog trace of what that fork did: committed `e56dab3` ("Fix: genericity …") onto branch `26-07-23`; `checkout` main; `cherry-pick` the same change onto main as `b332083`; branch switched back to `26-07-23`. i.e. it committed to a feature branch **and to trunk**, unreviewed.
- Concurrently, a consult fork spawned by *this* session (also forking `76ba13c6`) committed `8ca7bdb` to break its own `commit-guard` re-fire loop.

**Two distinct defects compound here:**

1. **Consult forks are not advisory — they are full autonomous agents.** `.claude/bin/consult` runs `claude -p --resume <id> --fork-session`. The Planner/Architect callback policy frames consults as "advisory, read-only; distill into the work doc." But the fork inherits full tool access and the invoking shell's cwd, so it can (and did) `git commit`/`checkout`/`cherry-pick`. Nothing sandboxes it to read-only.
2. **A shared session handle is forkable concurrently from multiple projects, into the wrong working tree.** `76ba13c6@play` is machine-local and both claude-workflow and orchestra resumed/forked it at once. The orchestra fork `cd`'d into claude-workflow. This directly violates the intended **one-way valve** of the orchestra spin-out (orchestra must not reach back into the parent) — see the "Orchestra handoff" memory.

**Impact.** Unreviewed commits landed on a feature branch (`e56dab3`) and on **trunk** (`b332083`); a session's working tree mutated between its own turns (phantom dirty `.claude/BASELINE.md`, `archive.md`, `initialize.md` appearing then vanishing); two sessions (this manager + an architect consult) were partly trapped in the `commit-guard` loop because the tree kept changing under them; risk of clobbering uncommitted work via the concurrent `git checkout main`/back.

**Where.** `.claude/bin/consult` (the `--fork-session` invocation and its cwd/tool inheritance); the cross-project handle-sharing is a machine-level concern (the `play` machine hosts both repos and both can resume `76ba13c6`). Related standing context: the orchestra one-way-valve intent; `.claude/agents/manager/CLAUDE.md` § Planner Callback (which asserts consults are advisory — the code doesn't enforce it).

**Note on `e56dab3`/`b332083`.** The genericity change itself is correct and on-mission (drops instance-history `work/` paths from the portable `.claude/` set) and was **kept** (operator-agreed 26-07-23). This bug is about the *mechanism*, not that commit's content. `b332083` sits directly on `main` — flagged for the operator's awareness.

**Needs** (for a future `/execute` — no resolution attempted here): likely an architect-tier design pass. Candidate directions to weigh, not decide: consult forks run read-only (strip write tools / run in a throwaway worktree or checkout); consult refuses when the resolved session's repo ≠ the invoking cwd's repo; the cross-project valve is enforced (a handle records its owning repo; foreign repos degrade to documents, never fork-in-place).
