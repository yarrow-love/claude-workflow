# `/initialize` check #7 carries project facts from another project

Filed 26-07-24, surfaced by an M3 phase-1 quality-gate run of the `initialize` audit. Not investigated — symptom and location only.

## Symptom

`.claude/commands/initialize.md:13` (check #7, Git hygiene) reads:

> 7. **Git hygiene** — `.gitignore` covers: secrets patterns (`.env*`, keys), large-format assets **(this repo: ephemeris data, catalogs, vendored clones)**, `tmp/`, `.codebase-memory/` …

The parenthetical is a project fact — and not even *this* project's. "Ephemeris data, catalogs, vendored clones" describes an astronomy repo; this repo carries no such assets. The phrase "this repo" inside a bootstrap file is self-contradicting: the file ships to every consuming repo.

This violates the standard root `CLAUDE.md` sets for the product: *"every file under `.claude/` must stay project-generic: no project facts, no machine-specific paths, no assumptions a fresh repo wouldn't satisfy. The seam for project specifics is `work/PROJECT.md` in the consuming repo."* Genericity is the stated acceptance test for `.claude/` changes, so this is a defect in the product, not a cosmetic nit.

## Where

- `.claude/commands/initialize.md:13` — the offending parenthetical.

## Note for whoever fixes it

The fix is presumably to make the asset class generic (e.g. "large binary/vendored assets, per `work/PROJECT.md`") rather than to delete the check. Worth a sweep for sibling leaks in the same file and its neighbors while there — this one survived the M2 coherence sweep because that sweep hunted retired-dispatch tokens, not project facts.
